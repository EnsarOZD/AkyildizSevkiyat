# Frontend Agent Analizi

## Madde Bazlı Değerlendirme

### Madde 5 — Modal Footer Butonları (KRİTİK)
**Sorun:** `py-2` → 32px yükseklik, WCAG 2.5.5 minimum 44px hedef boyutu ihlali
**Etki alanı:** Tüm modaller (global bileşen, her yerde kullanılıyor)
**Çözüm:**
```vue
<!-- components/ui/Modal.vue veya BaseModal footer slot -->
<footer class="flex gap-3 p-4">
  <button class="py-3 min-h-[44px] flex-1 ...">İptal</button>
  <button class="py-3 min-h-[44px] flex-1 ...">Onayla</button>
</footer>
```
**Dikkat:** `flex-1` sadece 2 butonlu footer'larda doğru çalışır. 3+ butonlu modal varsa `flex-1` yerine `min-w-[120px]` düşünülmeli.

---

### Madde 1 — MicroPickingModal / MacroPickingModal (KRİTİK)
**Sorun:** `h-full` + `h-10` input/buton, mobilde parmakla kullanılamaz (depo sahası kullanıcıları)
**Etki alanı:** 2 modal bileşen — aktif iş akışı, yanlış pick = stok hatası
**Çözüm:**
```vue
<!-- MicroPickingModal.vue & MacroPickingModal.vue -->
<input class="h-12 py-3 text-base ..." />
<button class="h-12 py-3 min-h-[44px] text-base ..." />
```
**Dikkat:**
- `h-full` modal container'da mobilde viewport taşması yapabilir → `max-h-[90vh] overflow-y-auto` eklenmeli
- `text-base` eklenmeli, iOS 16px altı inputlarda otomatik zoom yapar

---

### Madde 3 — ProjectMappingView Accessibility (YÜKSEK)
**Sorun:** Tablo içi `<select>` ve `<input>` elementlerinde `<label>` yok → screen reader erişilemez, WCAG 1.3.1 / 4.1.2 ihlali
**Çözüm:**
```vue
<!-- Görsel label istenmiyor ise sr-only pattern -->
<label :for="`project-select-${row.id}`" class="sr-only">
  Proje Seçimi - {{ row.name }}
</label>
<select :id="`project-select-${row.id}`" class="...">
  ...
</select>
```
**Dikkat:** `id` unique olmalı, `v-for` index değil `row.id` kullanılmalı. Mevcut tablo satır sayısı fazlaysa DOM'da çok sayıda label oluşur ama bu kabul edilebilir.

---

### Madde 2 — GoodsReceiptsView / PurchaseOrdersView Tabloları (ORTA)
**Sorun:** 9 sütun, `overflow-hidden`, iPad'de sütunlar kesiliyor
**Çözüm:**
```vue
<div class="overflow-x-auto">
  <table class="min-w-full">
    <th class="hidden lg:table-cell">Düşük öncelikli sütun</th>
    <!-- Kritik sütunlar her zaman görünür -->
    <th>Sipariş No</th>
    <th>Durum</th>
    <th>Tarih</th>
    <!-- Gizlenebilir sütunlar -->
    <th class="hidden lg:table-cell">Depo</th>
    <th class="hidden lg:table-cell">Açıklama</th>
  </table>
</div>
```
**Dikkat:**
- Hangi sütunların gizleneceği iş birimi ile teyit edilmeli
- `overflow-x-auto` eklendikten sonra sticky first-column düşünülebilir ama şu an scope dışı
- `overflow-hidden` kaldırılması parent layout'u etkileyebilir → test gerekli

---

### Madde 4 — EmptyState Spacing & Skeleton (DÜŞÜK)
**Sorun:** `py-16` mobilde ekranı boşaltıyor, skeleton satır sayısı hardcoded
**Çözüm:**
```vue
<!-- EmptyState.vue -->
<div class="py-8 sm:py-16 text-center">
  <slot />
</div>

<!-- Skeleton.vue -->
<script setup>
defineProps({
  rows: { type: Number, default: 5 }
})
</script>
<template>
  <div v-for="i in rows" :key="i" class="skeleton-row ..." />
</template>
```
**Dikkat:** `rows` prop eklenmesi mevcut tüm Skeleton kullanım noktalarında default'a düşecek → breaking change yok.

## Critical Issues

### 1. Madde 5 — Modal Footer Buton Sayısı Audit'i YAPILMADAN Uygulanmamalı
**Sebep:** Modal bileşeni global. `flex-1` eklenmesi, 3+ butonlu veya slot-based footer'larda layout kırılmasına neden olur. Frontend Agent bunu doğru tespit etmiş ve `min-w-[100px]` alternatifini önermiş. **Ancak audit yapılmadan hangi yaklaşımın seçileceği kesinleşemez.**

**Beklenen aksiyon:** Projede `grep -r "Modal" --include="*.vue"` ile tüm modal kullanım noktaları taranmalı, footer buton sayıları listelenmelidir. Bu liste olmadan Madde 5 kodu yazılmamalıdır.

### 2. Madde 2 — `overflow-hidden` Kaldırma Etkisi Doğrulanmamış
**Sebep:** `overflow-hidden` genellikle sadece scroll gizleme için değil, parent layout hesaplaması (flex/grid child overflow, absolute positioned element containment) için de kullanılır. Kaldırıldığında:
- Dropdown menüler veya tooltip'ler tablo dışına taşabilir (bu istenebilir — ama test edilmeli)
- Parent container'ın height hesaplaması değişebilir
- Sticky header veya footer varsa davranış değişebilir

**Beklenen aksiyon:** Değişiklik yapılmadan önce ilgili sayfada `overflow-hidden`'ın neden eklendiği commit history'den kontrol edilmeli veya en azından localhost'ta değişiklik yapılıp görsel kontrol geçilmelidir.

### 3. Madde 2 — Gizlenecek Sütunların İş Birimi Teyidi Eksik
**Sebep:** Frontend Agent "Depo", "Açıklama", "Birim" sütunlarını `hidden lg:table-cell` yapmayı önermiş ama bu varsayım. Yanlış sütun gizlenirse kullanıcılar kritik veri kaybı yaşar.

**Beklenen aksiyon:** PO (Product Owner) veya iş birimi ile hangi sütunların tablet/mobilde gizlenebileceği teyit edilmeli. Bu teyit olmadan sadece `overflow-x-auto` eklenmeli, sütun gizleme yapılmamalıdır.

## Edge Cases (Özetlenmiş)

- **E2 — iOS keyboard + 90vh:** Mobilde keyboard açıkken `90vh` hesaplaması sorunlu olabilir. `max-h-[90dvh]` veya JavaScript-based çözüm gerekebilir. Şu an `90vh` kabul edilebilir ama iOS test edilmeli.
- **E3 — row.id undefined:** Fallback olarak `row.id ?? index` kullanılmalı, aksi halde duplicate HTML id oluşur.
- **E6 — Modal footer slot:** Eğer modal bileşeninde footer slot varsa, dışarıdan gelen butonlar `min-h-[44px]` almaz. Bu slot pattern'inin kullanılıp kullanılmadığı audit'te kontrol edilmeli.

## Missing Checks

### Kodda Eksik

1. **Madde 3:** `row.id` undefined guard'ı yok. Frontend Agent kodu `row.id` varsayıyor ama veri kaynağında id olmayan satırlar olabilir.
   ```vue
   :id="`project-select-${row.id ?? index}`"
   ```

2. **Madde 5:** Modal footer'ın slot-based mi yoksa hardcoded mı olduğu belirsiz. Eğer slot kullanılıyorsa:
   ```vue
   <footer class="flex gap-3 p-4 [&>button]:min-h-[44px]">
     <slot name="footer" />
   </footer>
   ```
   Bu şekilde slot içinden gelen butonlara da min-height uygulanır.

3. **Madde 1:** `inputmode="numeric"` Frontend Agent'ın kodunda var ama orijinal talepte yok. Bu doğru bir ekleme (picking = sayısal veri) ama **mevcut inputlarda inputmode olup olmadığı kontrol edilmeli** — varsa override riski yok, yoksa davranış değişikliği olur (klavye tipi değişir).

4. **Madde 4:** Skeleton'da `role="status"` eklenmesi güzel ama **mevcut projede başka yerlerde de loading state varsa tutarlılık sağlanmalı**. Sadece Skeleton'a eklenmesi kısmen iyileştirme olur.

### Süreçte Eksik

5. **Görsel regression test aracı yok:** Bu 5 maddenin tamamı görsel değişiklik. Manuel kontrol yetersiz kalabilir. En azından değişiklik öncesi/sonrası screenshot karşılaştırması yapılmalı (Chromatic, Percy veya manual).

6. **Screen reader testi kim yapacak?** Madde 3 için VoiceOver veya NVDA testi planlanmış ama kimin yapacağı, hangi browser+SR kombinasyonunun test edileceği belirsiz.

## Simplifications

### Frontend Agent Analizi Hakkında

1. **Madde 3 — Hem label hem aria-label gereksiz olabilir:**
   Frontend Agent "ikisi birlikte en güvenli pattern" demiş. Teknik olarak doğru ama gereksiz redundancy. `<label for>` tek başına yeterlidir — screen reader `<label>` içeriğini zaten okur. `aria-label` yalnızca `<label>` kullanılamadığında gereklidir.

   **Öneri:** Sadece `<label for>` + `sr-only` yeterli. `aria-label` çıkarılabilir, kod temizlenir.

2. **Madde 2 — `-mx-4 sm:mx-0` karmaşıklık katıyor:**
   Bu edge-to-edge tablo görünümü için güzel ama mevcut padding yapısı bilinmeden eklemek layout kırabilir. İlk iterasyonda **sadece `overflow-x-auto`** yeterli, `-mx-4` ikinci iterasyonda değerlendirilebilir.

3. **Madde 4 — Skeleton validator üst limiti 20 keyfi:**
   `val <= 20` neden 20? Eğer bir tabloda 50 satır skeleton gösterilmek istenirse bu limit engelleyici olur. Validator'ı kaldırmak veya `val <= 50` yapmak daha güvenli. Ya da validator yerine sadece `val > 0` koşulu yeterli.
