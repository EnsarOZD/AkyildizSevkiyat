## 2. KULLANICI GRUPLARI — İŞ GERÇEKLİĞİ ANALİZİ

### 2.1 Şoför (Driver) — "Sistemin Saha Testi"

> **Tespit:** Şoför deneyimi bu projenin başarı/başarısızlık belirleyicisi.

**Neden bu kadar kritik:**
- Şoför sistemi kullanmazsa, dispatcher telefona döner. Telefona dönerse, sistem sadece "kayıt aracı" olur, "yönetim aracı" değil.
- Lojistik sektöründe en büyük dijitalleşme direnci şoförlerden gelir. UX kötüyse "ben eski sistemi kullanırım" denir ve tüm yatırım boşa gider.

**Gerçek Hayat Senaryoları (Tanımlanmamış Olanlar):**

| Senaryo | Soru | Neden Önemli |
|---------|------|-------------|
| Şoför tünelden geçiyor | Teslimat onayı internet olmadan atılabilir mi? | Bir gün boyunca 15-20 teslimat yapan şoför, her seferinde "bağlantı bekleniyor" görürse sistemi terk eder |
| Müşteri yerinde yok | "Teslimat yapılamadı" akışı var mı? Fotoğraf çekilebilir mi? Not eklenebilir mi? | Bu teslimatların %8-12'sinde yaşanır. Tanımlanmamış = dispatcher telefon açar |
| Kısmi teslimat | 10 kutudan 8'i teslim, 2'si hasarlı — bu nasıl kaydedilecek? | Muhasebe ve müşteri ilişkileri doğrudan etkilenir |
| Şoför yanlış adrese gitmiş | Rota değişikliği / adres düzeltme akışı var mı? | Gerçek hayatta haftada en az 1-2 kez olur |
| Araç bozulmuş | Sevkiyat başka şoföre nasıl devredilir? Sahadan mı, ofisten mi? | Operasyonel süreklilik |
| Müşteri imza atıyor | Dijital imza / fotoğraflı teslimat onayı var mı? | Hukuki ispat, müşteri şikayetlerinde delil |

> **Öneri:** Production'a almadan önce minimum bir şoförle bir gün sahada test yapılmalı. "Desk research" ile şoför UX'i çözülmez.

**Neden önemli:** Bu senaryoların her biri tanımlanmadığında, şoför dispatcher'ı arar → dispatcher işini bırakır → operasyon yavaşlar → yönetim "sistem işe yaramıyor" der.

### 2.2 Dispatcher — "Sistemin Kalp Atışı"

> **Tespit:** Dispatcher gün boyu sistemde yaşıyor. Her UI sürtünmesi çarpan etkisi yaratıyor.

**Sayılarla konuşalım:**

```
Günde ortalama sevkiyat sayısı (30 kişilik firma): ~40-80
Her sevkiyat için tıklama sayısı (tahmini):        ~8-15
Günlük toplam tıklama:                              ~320-1125
Her gereksiz tıklamanın maliyeti:                   ~2 saniye
Günlük kayıp (kötü UX):                            ~10-35 dakika
Aylık kayıp:                                        ~3.5-12 saat
```

**Tanımlanmamış Ama Olması Gereken Akışlar:**

| Akış | Durum | İş Etkisi |
|------|-------|-----------|
| Toplu sevkiyat oluşturma | Bilinmiyor | Pazartesi sabahı 40 sevkiyat tek tek mi girilecek? |
| Şoför-araç-rota otomatik eşleştirme önerisi | Muhtemelen yok | Dispatcher her seferinde düşünüp seçiyorsa, karar yorgunluğu |
| Sevkiyat şablonları (düzenli müşteriler) | Muhtemelen yok | Aynı müşteriye her hafta aynı sevkiyat → neden tekrar girilsin? |
| Drag-and-drop rota planlama | Muhtemelen yok | "Şu sevkiyatı şu şoföre sürükle" = çok sezgisel |
| Durum değişikliğinde bildirim | Bilinmiyor | Şoför "yüklendi" dediğinde dispatcher biliyor mu? |

> **Öneri:** Dispatcher ile 30 dakikalık bir "bir gününü anlat" görüşmesi yapılmalı. En çok ne zaman zorlanıyor, neyi tekrar tekrar yapıyor, hangi bilgiyi aramak zorunda kalıyor?

**Neden önemli:** Dispatcher'ın verimsizliği tüm zinciri etkiler. Şoförler bekler, müşteriler bekler, yönetim habersiz kalır.

### 2.3 Depo (Warehouse) — "Fiziksel Dünya ile Dijital Arayüz"

> **Tespit:** Bu kullanıcı grubu en "fiziksel" olanı. Bunu unutmak çok kolay.

**Gerçek Çalışma Koşulları:**
- **Sabah 05:00-07:00** → En yoğun. Hava karanlık veya loş depo.
- **Ayakta, hareket halinde** → Tablet bir yere yaslanmış veya elde
- **Eldivenli** → Parmak hassasiyeti düşük
- **Gürültülü ortam** → Sesli bildirim duyulmayabilir
- **Çoklu iş** → Hem fiziksel yükleme, hem sistem onayı

**Kritik Soru — Kimse Sormamış:**

> Depo personeli gerçekten tablet mı kullanıyor, yoksa masaüstü bir bilgisayardan mı giriyor?

Bu sorunun cevabı mimariyi değiştirir. Eğer depoda sabit bir masaüstü varsa, "büyük touch target" ihtiyacı azalır ama "hızlı geçiş" ihtiyacı artar. Eğer elde tablet varsa, tamamen farklı bir UX gerekir.

**Tanımlanmamış Akışlar:**

| Akış | Soru | Neden Önemli |
|------|------|-------------|
| Yükleme sırası | Hangi kamyon önce yüklenir? Sistem söylüyor mu? | Sabah kaosunda "hangisini yükleyeyim" sorusu |
| Yükleme tamamlama onayı | "Bu kamyon hazır" butonu var mı? | Dispatcher'ın bilgilendirilmesi |
| Malzeme eksik / hasarlı | Yükleme sırasında ürün hasarlıysa ne olur? | Sevkiyatın kısmen iptal / değişiklik akışı |
| Araç plakası / dorse eşleştirme | Doğru kamyona yükleme yapıldığını kim doğruluyor? | Yanlış kamyona yükleme = büyük operasyonel kayıp |

> **Öneri:** Depo operasyonu için ayrı, minimal bir "yükleme ekranı" düşünülmeli. Depo personelinin sevkiyat detaylarının tamamını görmesine gerek yok.

### 2.5 Muhasebe (Accounting) — "Doğruluk Her Şeydir"

> **Tespit:** Muhasebenin UX sorunu = veri kalitesi sorunu = para sorunu.

**BaseInput yokluğunun gerçek iş etkisi:**

```
50+ yerde farklı form input implementasyonu
→ Bazılarında validation var, bazılarında yok
→ Bazılarında format mask var (₺1.234,56), bazılarında yok
→ Muhasebeci "1234.56" yazar, sistem "1.234,56" bekler
→ Yanlış tutar kaydedilir
→ Ay sonunda mutabakat tutmaz
→ 2 gün ekstra çalışma
```

**Tanımlanmamış Gereksinimler:**

| Gereksinim | Durum | Neden Önemli |
|------------|-------|-------------|
| Sayı formatı tutarlılığı (₺) | BaseInput yok = risk | Finansal veri doğruluğu |
| Tarih formatı tutarlılığı | Bilinmiyor | GG.AA.YYYY mi, YYYY-MM-DD mi? |
| Ondalık separator (, vs .) | Bilinmiyor | TR'de virgül, JS'te nokta = hata potansiyeli |
| Düzenlenmiş alan geçmişi (audit trail) | Muhtemelen yok | "Kim, ne zaman, neyi değiştirdi?" = muhasebe güvenliği |
| Toplu fatura / irsaliye oluşturma | Bilinmiyor | Ay sonunda 200 fatura tek tek mi kesilecek? |

> **Öneri:** Muhasebe ekranlarında form validation production öncesi mutlaka standartlaştırılmalı. Yanlış finansal veri = düzeltmesi en pahalı hata türü.

## 3. TANIMLNMAMIŞ GEREKSİNİMLER VE "YA ŞU OLURSA?" ANALİZİ

### 3.1 Genel Sistem Senaryoları

|---|---------|-------------|-----------|
| 1 | İki dispatcher aynı anda aynı sevkiyatı düzenlerse ne olur? | ❌ Hayır | Veri kaybı / çakışma |
| 2 | Şoför sahada telefonunu kaybederse, oturumu başka cihazdan açabilir mi? | ❌ Hayır | Güvenlik + operasyonel süreklilik |

# Akyıldız Sevkiyat — İş Analizi Raporu

## Yönetici Özeti

Bu frontend iyileştirme planını iş perspektifinden incelediğimde, teknik açıdan oldukça kapsamlı ve iyi düşünülmüş bir çalışma görüyorum. Production'a yakın bir sistemde 50+ dosyayı etkileyen bir migration, iş sürekliliği açısından ciddi risk taşıyor.

Aşağıda, her iş fonksiyonu ve kullanıcı grubu için tespitlerimi, bunların neden önemli olduğunu ve somut önerilerimi sunuyorum.

## 1. Gereksinim Bütünlüğü Analizi

### 1.1 "Asıl Problemi Çözüyor mu?" Sorusu

**Tespit:** Plan, UI tutarsızlığını (focus ring renkleri, label boşlukları, inline SVG'ler) birincil sorun olarak ele alıyor. Ancak 30 kişilik bir lojistik şirketinin sabah 05:00'te karşılaştığı gerçek sorun bu mu?

**Neden önemli:** Sahada bir şoför "focus ring'in rengi tutarsız" diye şikayet etmiyor. Şoför şunu söylüyor: "İnternetsiz yerdeyken teslimatı onaylayamıyorum, sonra unutuyorum." Depo personeli şunu söylüyor: "Yüklemeyi onaylarken hangi ürünü hangi kamyona koyacağımı bulmak 3 dakika sürüyor." Dispatcher şunu söylüyor: "Bugün 47 sevkiyat var, hangisinde sorun olduğunu anlamak için hepsini tek tek açmam gerekiyor."

**Öneri:** Plan, sorunları teknik önem sırasına koymuş ama **iş etkisi sırasına koymalı.** Şu sıralama daha doğru:

| Sıra | Sorun | İş Etkisi |
|------|-------|-----------|
| 1 | Şoför offline çalışamıyor | **Gelir kaybı** — teslimat onaylanmıyor, fatura kesilemiyor |
| 2 | Dispatcher günlük özet göremiyor | **Verimlilik kaybı** — günde 30-60 dk boşa gidiyor |
| 3 | Depo tablet deneyimi zayıf | **Operasyonel risk** — yanlış yükleme, iade |
| 4 | Muhasebe sayı formatı tutarsız | **Mali risk** — yanlış tutar = hatalı fatura |
| 5 | BaseInput tutarsızlığı | **Geliştirici verimliliği** — yeni ekran 2x uzun sürüyor |
| 6 | Design token eksikliği | **Bakım maliyeti** — gelecek değişiklikler zor |

### 1.2 Tanımlanmamış İş Gereksinimleri

**Tespit:** Planda 5 kullanıcı grubunun akışları tarif edilmiş ama şu kritik iş kuralları hiçbir yerde yok:

**Neden önemli:** Bunlar production'da ilk hafta patlayacak şeyler.

**Eksik gereksinimler listesi:**

> **a) Sevkiyat Durumu Geçiş Kuralları**

> - Durum geçişi için onay mekanizması var mı? Mesela 50.000 TL üzeri bir sevkiyatın iptali manager onayı mı gerektiriyor?

> **b) Eşzamanlı Erişim**

## 2. Kullanıcı Grubu Bazlı Analiz

### 2.1 Şoför (Driver) — En Kritik Kullanıcı

**Tespit:** Plan, offline durumu "detection" olarak tanımlamış ama offline **çalışma stratejisi** tanımlamamış. "İnternetin yok" uyarısı göstermek çözüm değil; şoför zaten interneti olmadığını biliyor.

**Neden önemli:** Bir kamyon şoförü günde ortalama 8-15 teslimat yapıyor. Her teslimat noktasında 2-3 dakika uygulama ile etkileşim var. Eğer teslimatların %10'unda internet yoksa ve şoför onay yapamıyorsa:
- Günde 1-2 teslimat "onaysız" kalıyor
- Dispatcher bunu takip edemiyor
- Muhasebe faturayı kesemiyor
- Müşteri "teslim edilmedi" diyor

Bu, **ayda potansiyel olarak 30-60 teslimatın belirsiz durumda kalması** demek.

**Öneri:**
- Offline-first yaklaşım şart. Şoför offline iken "Teslim Edildi" butonuna basabilmeli, veri yerel olarak saklanmalı, internet geldiğinde otomatik senkronize olmalı.
- Planda "offline detection" var ama "offline queue" yok. Bu, iş açısından yarım bir çözüm.
- **Minimum Viable Offline:** En azından teslimat durumu güncelleme ve konum kaydı offline çalışmalı. Yeni sevkiyat oluşturma offline olmak zorunda değil.
- Senkronizasyon çakışması kuralları tanımlanmalı: Şoför offline iken dispatcher sevkiyatı iptal ettiyse, şoför "Teslim Edildi" dediğinde ne olacak?

---

### 2.2 Depo Personeli (Warehouse) — Göz Ardı Edilen Kullanıcı

**Tespit:** Planda "touch target boyutları tutarsız, sabah 05:00'te eldivenle küçük butonlara basmak zor" tespiti yapılmış. Bu doğru ama çok yüzeysel.

**Neden önemli:** Depo operasyonu günün en zaman-kritik operasyonu. Kamyonlar saat 06:00-07:00 arasında çıkmalı. Her dakika gecikme = rota bozulması = müşteri memnuniyetsizliği.

**Eksik gereksinimler:**
- **Barkod/QR okuma:** Depo personeli malzemeleri manuel mi kontrol ediyor? Tablet kamerası ile barkod okuma var mı? Bu, yükleme hatasını %90 azaltır.
- **Yükleme sırası optimizasyonu:** Kamyona hangi malzeme önce yüklenmeli? Son teslimat noktasının malzemesi ilk yüklenmeli ki alttan çıkarmak gerekmesin. Bu bilgi ekranda var mı?
- **Sesli/görsel geri bildirim:** Eldivenle tablet kullanıyorken, başarılı/başarısız işlem için güçlü bir görsel ve opsiyonel haptic/ses geri bildirimi gerekiyor. Sadece renk değişimi yetmez — yeşil/kırmızı renk körü %8 erkek kullanıcıda çalışmaz.
- **Büyük font modu:** Depo loş olabilir, tablet parlaklığı düşük olabilir. Normal 14px metin okunmayabilir. Depo kullanıcıları için otomatik olarak daha büyük font boyutu uygulanmalı.

---

### 2.3 Dispatcher — Verimlilik Kaybı

**Tespit:** Planda dispatcher'ın günlük özet göremediği söyleniyor. ShipmentDetailView'ın 1,466 satır olması teknik bir sorun olarak sunulmuş.

**Neden önemli:** Dispatcher'ın asıl sorunu bilgi hiyerarşisi. 47 sevkiyatı yönetirken şu bilgiye 2 saniyede ulaşması gerekiyor:
- Kaç tanesi planlandığı gibi gidiyor?
- Kaç tanesi gecikmeli?
- Kaç tanesi sorunlu (müşteri yok, adres yanlış, vb.)?

**Planda eksik olan kritik akışlar:**
- **Toplu işlem (bulk action):** Dispatcher 10 sevkiyatı aynı şoföre atamak istiyorsa, tek tek mi yapacak? Toplu seçim + toplu atama lazım.
- **Hızlı durum filtresi:** "Sadece sorunluları göster" bir tık olmalı. Mevcut planda filtreleme UX'i tarif edilmemiş.
- **Bildirim merkezi:** Şoförden gelen güncellemeler, müşteri talepleri, gecikme uyarıları... Dispatcher bunları nereden takip ediyor? Ayrı bir bildirim paneli var mı?
- **Harita görünümü:** 47 sevkiyatın nerede olduğunu haritada görmek, dispatcher için tablo kadar önemli. Plan harita bileşeninden hiç bahsetmiyor.

---

### 2.4 Muhasebe — Mali Risk

**Tespit:** Planın Flow 5'inde "sayı formatı tutarsızlığı" tespiti doğru ve kritik.

**Neden önemli:** Bu sadece UX sorunu değil, **mali hata riski.** 1.234,56 TL ile 1.23456 TL arasındaki fark = 1.233 TL hata. Günde 30 faturada bu tür hatalar birikirse, ayda ciddi tutarlar oluşur. Ayrıca e-fatura/e-arşiv entegrasyonu varsa, format uyumsuzluğu faturanın reddedilmesine neden olur.

**Planda eksik olanlar:**
- Tutar alanlarında otomatik binlik separator gösterilmeli (input masking)

```vue
<template>
  <div :class="wrapperClass">
    <label
      v-if="label"
      :for="inputId"
      class="block text-sm font-medium text-gray-700 dark:text-gray-300 mb-label-gap"
    >
      {{ label }}
      <span v-if="required" class="text-danger-500 ml-0.5">*</span>
    </label>

    <div class="relative">
      <!-- Left Icon -->
      <div
        v-if="$slots.iconLeft"
        class="absolute inset-y-0 left-0 pl-3 flex items-center pointer-events-none text-gray-400"
      >
        <slot name="iconLeft" />
      </div>

      <input
        :id="inputId"
        ref="inputRef"
        v-bind="$attrs"
        :value="modelValue"
        :type="type"
        :placeholder="placeholder"
        :disabled="disabled"
        :readonly="readonly"
        :class="[
          inputClasses,
          { 'pl-10': $slots.iconLeft },
          { 'pr-10': $slots.iconRight || error },
        ]"
        @input="$emit('update:modelValue', $event.target.value)"
        @blur="$emit('blur', $event)"
        @focus="$emit('focus', $event)"
      />

      <!-- Right Icon / Error Icon -->
      <div
        v-if="$slots.iconRight || error"
        class="absolute inset-y-0 right-0 pr-3 flex items-center"
      >
        <ExclamationCircleIcon v-if="error" class="h-5 w-5 text-danger-500" />
        <slot v-else name="iconRight" />
      </div>
    </div>

    <!-- Hint / Error Text -->
    <p v-if="error" class="mt-1 text-sm text-danger-600 dark:text-danger-400">
      {{ error }}
    </p>
    <p v-else-if="hint" class="mt-1 text-caption text-gray-500 dark:text-gray-400">
      {{ hint }}
    </p>
  </div>
</template>

<script setup>
import { computed, useId, ref } from 'vue'
import { ExclamationCircleIcon } from '@heroicons/vue/20/solid'

defineOptions({ inheritAttrs: false })

const props = defineProps({
  modelValue:  { type: [String, Number], default: '' },
  label:       { type: String, default: '' },
  type:        { type: String, default: 'text' },
  placeholder: { type: String, default: '' },
  hint:        { type: String, default: '' },
  error:       { type: String, default: '' },
  disabled:    { type: Boolean, default: false },
  readonly:    { type: Boolean, default: false },
  required:    { type: Boolean, default: false },
  size:        { type: String, default: 'md', validator: v => ['sm', 'md', 'lg'].includes(v) },
})

defineEmits(['update:modelValue', 'blur', 'focus'])

const inputRef = ref(null)
const inputId = useId()

const sizeMap = {
  sm: 'px-2.5 py-1.5 text-sm',
  md: 'px-3 py-2 text-sm',
  lg: 'px-4 py-3 text-base',
}

const inputClasses = computed(() => [
  // Base
  'block w-full rounded-input border transition-colors duration-150',
  'bg-white dark:bg-surface-2',
  'text-gray-900 dark:text-gray-100',
  'placeholder:text-gray-400 dark:placeholder:text-gray-500',
  // Size
  sizeMap[props.size],
  // State
  props.error
    ? 'border-danger-300 dark:border-danger-600 focus:ring-danger-500 focus:border-danger-500'
    : 'border-gray-300 dark:border-gray-600 focus:ring-brand-500 focus:border-brand-500',
  // Disabled
  props.disabled && 'opacity-50 cursor-not-allowed bg-gray-50 dark:bg-surface-3',
  // Focus ring — TUTARLI (blue/indigo sorununu çözer)
  'focus:outline-none focus:ring-2 focus:ring-offset-0',
])

const wrapperClass = computed(() => props.size === 'lg' ? 'space-y-2' : 'space-y-label-gap')

// Expose for programmatic focus
defineExpose({ focus: () => inputRef.value?.focus() })
</script>
