# Manager Agent — Rota Optimizasyon Sayfası

---

## Task Summary
Yeni bir "Rota Optimizasyon" sayfası oluşturulacak. Kullanıcı proje kodları girer; sistem ISS API ile senkronizasyon kontrolü yapar (fark varsa onay ister), ardından güncel adresler üzerinden optimum teslimat rotası hesaplar.

## Task Type
**Feature** (yeni sayfa + iki alt özellik)

## Affected Areas
- **Backend** — DTO değişikliği, yeni endpoint'ler, rota hesaplama servisi, ISS karşılaştırma mantığı
- **Frontend** — yeni sayfa, bileşenler, state yönetimi, API entegrasyonu
- **Database** — mevcut şema yeterli (yeni tablo/kolon gerekmiyor, Address zaten entity'de var)
- **Integration** — ISS API (mevcut), Google Maps Directions/Routes API (yeni)
- **Auth** — mevcut auth yeterli (yeni rol/yetki gerekmeyebilir ama doğrulanmalı)

## Selected Agents
1. **Backend Agent** — DTO düzeltmesi, yeni endpoint'ler, ISS diff servisi, rota optimizasyon servisi
2. **Frontend Agent** — yeni sayfa, bileşenler, akış, state
3. **Architect Agent** — cross-layer impact (ISS + harici rota API entegrasyonu, DTO değişikliği frontend'i etkiler, koordinat kaynağı belirsizliği mimari karar gerektirir)
4. **QA-Reviewer Agent** — edge case'ler, validasyon, riskler, son kritik

## Execution Plan

### Phase 0 — Architect (önce çalışmalı, kararlar diğer agent'ları etkiler)

**Mimari Kararlar:**

1. **Koordinat Kaynağı Stratejisi**
   - Mevcut Latitude/Longitude → şoför GPS'inden geliyor
   - ISS'ten koordinat gelmiyor
   - Rota optimizasyonu için adres → koordinat dönüşümü (geocoding) gerekiyor
   - **Karar:** Rota hesaplama sırasında adres bazlı çalışılmalı. Google Maps Directions API zaten adres kabul eder, geocoding ayrıca gerekmeyebilir. Ancak cache'leme için geocode edilmiş koordinatlar ayrı bir alana (RouteLatitude/RouteLongitude) kaydedilebilir — ya da mevcut Lat/Lng alanları karıştırılmamalı (şoför GPS ≠ adres koordinatı)
   - **Öneri:** Mevcut Lat/Lng alanlarına dokunma. Rota API'ye adres string olarak gönder. İleride geocoding gerekirse ayrı alan eklensin.

2. **Rota API Soyutlama**
   - Hangi API kullanılacağı belirsiz → **Strategy Pattern** ile soyutlanmalı
   - `IRouteOptimizationService` interface'i tanımlanmalı
   - İlk implementasyon: Google Maps Directions API (waypoint optimization destekliyor)
   - Alternatifler: Google Routes API, OSRM, GraphHopper

3. **Akış Mimarisi**
   ```
   Kullanıcı proje kodları girer
         ↓
   Backend: ISS'ten güncel veri çek + mevcut veriyle karşılaştır
         ↓
   Frontend: diff'leri göster → kullanıcı onaylar/reddeder
         ↓
   Backend: onaylananları güncelle
         ↓
   Backend: güncel adreslerle rota optimizasyonu hesapla
         ↓
   Frontend: sıralı rota listesini göster
   ```

4. **DTO Değişikliği Etki Analizi**
   - Address'in ProjectDto'ya eklenmesi mevcut frontend'i bozmamalı (additive change)
   - Yeni bir `ProjectSyncComparisonDto` oluşturulmalı (diff için)
   - Yeni bir `RouteOptimizationResultDto` oluşturulmalı

5. **API Yapısı Önerisi**
   - `POST /api/route-optimization/compare` → ISS diff döner
   - `POST /api/route-optimization/sync` → onaylanan güncellemeleri uygular
   - `POST /api/route-optimization/optimize` → rota sırasını hesaplar
   - Üçü ayrı endpoint olmalı (kullanıcı onayı arada gerçekleşiyor)

### Phase 2 — QA-Reviewer (son aşama)

**Validasyon Kontrolleri:**

1. **Boş/geçersiz proje kodu** → backend'de validate, frontend'de de ön kontrol
2. **ISS'te bulunmayan proje kodu** → karşılaştırma sonucunda "ISS'te bulunamadı" durumu eklenmeli
3. **Adresi olmayan proje** → rota hesaplama öncesi filtrelenmeli, kullanıcıya bildirilmeli
4. **Google Maps waypoint limiti (25)** → 25'ten fazla proje varsa backend'de hata veya bölme stratejisi
5. **Aynı adresli projeler** → rota hesaplamada sorun olmaz ama kullanıcıya bilgi verilebilir
6. **Concurrent güncelleme** → iki kullanıcı aynı projeyi güncellerse? → optimistic concurrency veya last-write-wins (mevcut davranışa uygun)
7. **ISS API timeout** → timeout süresi yapılandırılabilir olmalı, retry politikası
8. **Google Maps API quota/billing** → rate limiting, quota aşımı durumunda anlamlı hata mesajı
9. **Kısmi onay senaryosu** → kullanıcı sadece bazı değişiklikleri onaylarsa, rota hesaplama güncel+eski adres karışımı kullanır — bu beklenen davranış mı? → **Evet, kullanıcı kararına bırakılmalı**
10. **Address alanı null/empty olan projeler** → ISS'ten de adres gelmiyorsa rota hesaplamaya dahil edilmemeli

**Güvenlik:**
- Google Maps API key backend'de kalmalı, frontend'e sızmamalı
- ISS API key mevcut güvenlik modeline uymalı
- Yeni endpoint'lere mevcut auth uygulanmalı

**Performans:**
- ISS karşılaştırma: projeler paralel çekilebilir (batch mümkünse batch)
- Google Maps API çağrısı tek request (waypoint olarak)

```csharp
public class GoogleMapsRouteOptimizationService : IRouteOptimizationService
{
    private readonly HttpClient _httpClient;
    private readonly RouteOptimizationOptions _options;
    private readonly ILogger<GoogleMapsRouteOptimizationService> _logger;

    public async Task<RouteOptimizationResultDto> OptimizeRouteAsync(
        RouteOptimizationRequestDto request, CancellationToken ct)
    {
        // 1. DB'den projectCodes ile projeleri çek → adreslerini al
        // 2. Address null/empty olanları filtrele, loglama yap
        // 3. MaxWaypoints kontrolü (25 limit) → aşarsa hata fırlat
        // 4. Google Directions API çağrısı:
        //    - origin: startAddress ?? ilk adres
        //    - destination: son adres (veya origin'e dönüş)
        //    - waypoints: optimize:true|adres1|adres2|...
        // 5. Response'tan waypoint_order dizisini al → sıralama
        // 6. Her leg'den distance/duration bilgilerini çek
        // 7. RouteOptimizationResultDto oluştur ve döndür
    }
}
```

**Google Directions API Request Format:**
```
GET /maps/api/directions/json
  ?origin=DEPO_ADRESI
  &destination=DEPO_ADRESI (veya son durak)
  &waypoints=optimize:true|ADRES_1|ADRES_2|ADRES_3
  &key=API_KEY
  &language=tr
```

**Response'tan kullanılacak alanlar:**
- `routes[0].waypoint_order` → optimum sıralama indeks dizisi
- `routes[0].legs[i].distance.text` → mesafe
- `routes[0].legs[i].duration.text` → süre

## Required Conditions

### 1. Sync Aşamasında ISS Değerlerinin Frontend'den Gelmesi Reddedilmeli

Backend analizi şunu öneriyor: *"Frontend compare sonucundaki ISS değerlerini sync request'ine dahil etsin"* — yani `SyncApprovalRequestDto`'ya `IssName` ve `IssAddress` eklenmesi.

**Bu kabul edilemez.** Frontend'den gelen ISS değerlerinin doğrudan DB'ye yazılması:
- Güvenlik açığı: client, ISS'ten gelmeyen keyfi değerleri gönderebilir
- Veri bütünlüğü sorunu: compare ile sync arasında frontend manipülasyonuna açık
- Backend'in "ISS'ten gelen veri" garantisi ortadan kalkar

**Gerekli çözüm:** Sync aşamasında backend ISS'ten tekrar çekmeli. Ekstra API call maliyeti kabul edilebilir — veri bütünlüğü önceliklidir. Alternatif olarak compare sonucu backend'de kısa ömürlü cache'lenebilir (in-memory, 5-10 dk TTL, kullanıcı bazlı key). Ancak cache yoksa bile ISS'ten tekrar çekmek doğru mimari karardır.

### 2. IRouteOptimizationService İçinden Repository Erişimi Kaldırılmalı

Backend analizinde `GoogleMapsRouteOptimizationService` doğrudan DB'den proje çekiyor (*"DB'den projectCodes ile projeleri çek → adreslerini al"*). Infrastructure katmanındaki bir servis doğrudan repository'ye erişmemeli.

**Gerekli çözüm:** Controller veya Application katmanındaki bir orkestrasyon servisi (veya doğrudan controller) projeleri DB'den çeksin, adresleri hazırlasın, `IRouteOptimizationService.OptimizeRouteAsync`'e sadece adres listesi göndersin. Route optimization servisi yalnızca adres → rota hesaplama sorumluluğunu taşımalı. Bu ayrım aynı zamanda Strategy Pattern'in amacına da hizmet eder — farklı provider'lar sadece adres alıp rota döner.

**Somut etki:**
```
// YANLIŞ (şimdiki plan):
IRouteOptimizationService.OptimizeRouteAsync(projectCodes, startAddress)
  → repository'den proje çek → adresleri al → API çağır

// DOĞRU:
Controller/AppService → repository'den projeleri çek → adresleri hazırla
IRouteOptimizationService.OptimizeRouteAsync(addresses, startAddress)
  → sadece API çağır → sıralama döndür
```

`RouteOptimizationRequestDto` buna göre revize edilmeli: `ProjectCodes` yerine adres listesi almalı, ya da ayrı bir internal DTO kullanılmalı.

### 3. Compare Endpoint'inin ISS API Davranışı Netleştirilmeli

Mevcut `SyncProjects` tüm projeleri toplu çekiyor gibi görünüyor. Yeni compare servisi proje kodlarına göre filtrelenmiş çekim yapacak. **Mevcut ISS API client'ı bunu destekliyor mu?**

- ISS API proje koduna göre filtreleme destekliyorsa → doğrudan kullanılabilir
- Desteklemiyorsa → tüm projeleri çekip client-side filtreleme yapılmalı, bu da performans riski taşır
- **Bu sorunun cevabı backend implementasyonundan önce netleşmeli**, çünkü servislerin iç yapısını değiştirir

### 4. Google Maps API Key Yönetimi

`appsettings.json`'da API key tutulması üretim ortamı için uygun değil.

**Gerekli:** Üretim ortamında environment variable, Azure Key Vault, veya benzeri secret management kullanılmalı. Development'ta `appsettings.Development.json` veya user-secrets kabul edilebilir. Bu kararın kod yazılmadan önce alınması gerekiyor ki `RouteOptimizationOptions` bind'ı buna göre yapılsın.

### 5. Rota Dönüş Tipi: Açık Uçlu mı Kapalı Döngü mü?

Plan bunu belirsiz bırakıyor. Google Directions API'de:
- `origin ≠ destination` → açık rota (A→B→C)
- `origin = destination` → kapalı döngü (depo→B→C→depo)

**Karar verilmeli:** Teslimat senaryosunda dönüş var mı? Bu, frontend'in başlangıç/bitiş adresi UI'ını ve backend'in API çağrısını doğrudan etkiler. Varsayılan davranış açıkça belgelenmeli.

## Risks

### Orta — ISS API'ye Çifte Bağımlılık
Compare ve sync aşamalarında ISS API'ye iki ayrı çağrı yapılması (Condition #1 gereği) toplam yanıt süresini artırır. ISS API yavaş/güvenilmezse kullanıcı deneyimi kötüleşir. **Mitigation:** Compare sonucunu backend'de kısa ömürlü cache'le. Cache varsa sync sırasında cache'ten oku, yoksa tekrar çek.

### Orta — Google Maps API Maliyet Kontrolsüzlüğü
Rate limiting planlanmış ancak kullanıcı bazlı veya toplam günlük limit tanımlanmamış. 25 waypoint'li bir Directions API çağrısı standart bir çağrıdan pahalı. **Mitigation:** Günlük/saatlik call limit'i konfigürasyonda tanımlanmalı, aşıldığında kullanıcıya "limit aşıldı" mesajı gösterilmeli.

### Düşük-Orta — Mevcut SyncProjects Refactoring Riski
Fetch mantığının extract edilmesi basit bir refactoring ama mevcut `SyncProjects` endpoint'inin davranışını bozmama garantisi gerekli. **Mitigation:** Refactoring öncesi mevcut davranışın entegrasyon testi yazılması önerilir.

### Düşük — ProjectDto Address Eklenmesi
Additive change, mevcut consumer'ları bozmamalı. Ancak eğer herhangi bir frontend bileşeni `Object.keys(project)` gibi bir yapı kullanıyorsa beklenmeyen davranış olabilir. **Mitigation:** Mevcut proje listesi sayfasının kısa smoke test'i yeterli.

- Frontend'in başlangıç/bitiş adresi UI'ını etkiler
- Rota mesafe/süre hesabını değiştirir

Plan bunu tamamen belirsiz bırakıyor. "Opsiyonel başlangıç adresi" var ama bitiş noktası ne? Kullanıcı başlangıç adresi girmezse `origin` ne olacak?

**Çözüm:** Varsayılan davranış açıkça tanımlanmalı. Öneri:
- `startAddress` verilirse → origin ve destination bu adres olsun (kapalı döngü, depoya dönüş mantıklı)
- `startAddress` verilmezse → açık rota, ilk proje origin, son proje destination
- Frontend'de bu davranış kullanıcıya açıkça belirtilmeli

## Edge Cases

- **ISS API kısmi yanıt:** 10 projeden 7'sinin ISS verisi döner, 3'ü timeout. Bu durumda başarılı olanlar gösterilmeli, başarısız olanlar "ISS verisi alınamadı" olarak işaretlenmeli. Toptan hata yerine partial success desteklenmeli.

- **Google Maps API'nin waypoint_order döndürmemesi:** API hata döndürmeden başarılı yanıt verir ama optimizasyon yapamaz (nadir). `waypoint_order` null/empty kontrolü yapılmalı, fallback olarak orijinal sıra dönebilir.

- **Rota hesaplama sırasında adres geocode edilemez:** Google Maps adresi tanıyamaz → `ZERO_RESULTS` status döner. Bu proje rota dışı bırakılmalı ve kullanıcıya bildirilmeli. Tüm rota fail etmemeli.

- **Cache TTL sırasında proje silinir:** Compare cache'te duran bir proje DB'den silinirse, sync aşamasında "proje bulunamadı" hatası dönmeli.

- **Tarayıcı geri butonu:** Wizard Aşama 3'teyken geri basılırsa ne olur? State kaybolur mu? Browser history ile wizard step senkronizasyonu düşünülmeli veya en azından "verileriniz kaybolacak" uyarısı.

## Missing Checks

### Backend
1. **Proje kodlarının DB'de varlık kontrolü:** Compare endpoint'inde ISS'te bulunamayan kodlar handle ediliyor ama DB'de hiç olmayan kodlar ne olacak? Ayrı bir `notFoundInDb` listesi de dönmeli.
2. **Rate limiting:** Plan "IP/user başına dakikada max 5 request" diyor ama hangi endpoint'e? Sadece optimize endpoint'ine mi, yoksa tüm route-optimization endpoint'lerine mi? Optimize endpoint'i yeterli — compare ve sync ISS API'ye gider ama maliyet yaratmaz.
3. **Google Maps API response validation:** API'den dönen JSON'un beklenen yapıda olduğunun strict kontrolü. `routes` boş dizi dönebilir, `legs` eksik olabilir. Defensive parsing şart.
4. **Logging:** Her API çağrısı (ISS + Google Maps) request/response loglanmalı. Özellikle Google Maps çağrılarında maliyet takibi için zorunlu.
5. **CancellationToken propagation:** Tüm async zincirlerde CancellationToken iletilmeli. Kullanıcı sayfadan çıkarsa gereksiz API çağrısı devam etmemeli.

### Frontend
6. **Aşamalar arası veri kaybı uyarısı:** Aşama 2'deyken "Yeni Rota" butonuna basılırsa veya sayfadan çıkılırsa confirm dialog gösterilmeli.
7. **Retry mekanizması:** Her aşamada "Tekrar Dene" butonu planlanmış ama retry count limiti yok. Sonsuz retry döngüsü önlenmeli (3 deneme sonra farklı mesaj).
8. **Erişilebilirlik (a11y):** Tag input bileşeninde aria-label, tablo'da scope attribute, step indicator'da aria-current. Minimum seviyede bile olsa düşünülmeli.

## Simplifications

### 1. Compare + Sync Birleştirilebilir mi?
Hayır. Kullanıcı onay adımı arada olduğu için iki ayrı endpoint doğru. Ancak **fark yoksa sync endpoint'ine hiç gidilmemeli** — frontend bunu "fark yok → doğrudan optimize" ile handle ediyor, doğru.

### 2. StepIndicator Gerekli mi?
3 aşamalı basit akış için ayrı bir bileşen yerine sayfa içinde basit bir progress bar yeterli olabilir. Ancak UX açısından bileşen yaklaşımı daha iyi, karmaşıklık maliyeti düşük. **Kabul edilebilir.**

### 3. Cache Katmanı Gerçekten Gerekli mi?
Architect cache önermiş ama scope'u küçük tutmak için: ilk iterasyonda cache olmadan, sync aşamasında ISS'ten tekrar çekmek yeterli. Cache, performans sorunu yaşanırsa ikinci iterasyonda eklenebilir. **Bu basitleştirme yapılmalı** — premature optimization'dan kaçınılmalı.

### 4. ExcludedProjects Ayrı Bir DTO Alanı mı Olmalı?
Evet. Adresi eksik veya geocode edilemeyen projelerin neden hariç tutulduğu kullanıcıya gösterilmeli. Backend'in `RouteOptimizationResultDto`'da `excludedProjects` alanı doğru. Frontend analizinde bu alan tip tanımında var ama kullanım detayı eksik — `RouteResultList`'te uyarı banner'ında gösterilmeli.

## Recommendation

### Devam Etmeden Önce Çözülmesi Gerekenler (Blockers)

1. **SyncApprovalRequestDto'dan IssName/IssAddress kaldırılmalı.** Backend sync aşamasında ISS'ten tekrar çekmeli. Cache opsiyonel (ikinci iterasyona bırakılabilir).

2. **IRouteOptimizationService interface'i revize edilmeli.** Repository erişimi orkestrasyon katmanına taşınmalı, servis sadece adres listesi almalı.

3. **Açık/kapalı rota kararı verilmeli.** Teslimat senaryosunda depoya dönüş var mı? Bu karar olmadan backend implementasyonu doğru yapılamaz.

4. **ISS API'nin proje koduna göre filtreleme yapıp yapamadığı netleştirilmeli.** Tüm projeleri çekip client-side filtreleme gerekliyse, performans etkisi değerlendirilmeli ve compare endpoint'inin yanıt süresi beklentisi buna göre ayarlanmalı.
