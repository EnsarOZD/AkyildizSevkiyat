## Domain 1 — Login & JWT Auth

### Files (Likely Affected)
- `AuthController` (login, logout, refresh endpoints)
- `JwtService` / `TokenService` (token generation, validation, refresh)
- `RefreshToken` model/entity + migration
- `AuthMiddleware` / `JwtGuard` (request authentication)
- `RateLimiter` middleware (login endpoint)
- `UserService` (password validation, account lockout)
- `PermissionMiddleware` / `RoleGuard` (authorization layer)
- `TenantScope` / `TenantMiddleware` (multi-tenancy isolation)

### Changes / Findings

**API:**
```
CRITICAL — Token Refresh & Revocation
├── Refresh token rotation: When a refresh token is used, the old 
│   one MUST be invalidated and a new one issued. Without this, 
│   a leaked refresh token grants permanent access.
│   → Verify: RefreshToken table exists with columns: 
│     [id, user_id, token_hash, expires_at, revoked_at, replaced_by_id]
│   → If rotation is missing: single biggest auth vulnerability.
│
├── Concurrent refresh race condition: Two tabs simultaneously 
│   send refresh requests. Tab A rotates token, Tab B's token is 
│   now invalid → Tab B gets 401 → user logged out unexpectedly.
│   → Fix: Grace period (e.g., old token valid for 10s after rotation)
│   OR family-based revocation (revoke entire token family on reuse).
│
├── Logout revocation: Logout endpoint must set revoked_at on the 
│   refresh token. If tokens are stateless JWT only with no DB 
│   record → logout is cosmetic, token still works until expiry.
│   → Verify: POST /auth/logout invalidates server-side state.
│
└── Access token blacklisting: For immediate permission revocation 
    (e.g., user deactivated by admin), either:
    (a) Short access token TTL (5-15 min) + refresh check, OR
    (b) Redis-based blacklist checked in AuthMiddleware
    → Current state unknown — must verify.
```

```
HIGH — Brute Force & Account Security
├── Rate limiting on POST /auth/login:
│   ├── Must be account-based (not just IP) to prevent 
│   │   distributed brute force
│   ├── Lockout after N failures (e.g., 5 attempts → 15 min lock)
│   └── Lockout notification to user/admin?
│
├── Password complexity enforcement:
│   → MUST be validated server-side regardless of frontend rules
│   → Minimum: 8 chars, 1 upper, 1 lower, 1 digit (or equivalent policy)
│   → Verify: UserService or password validation rule exists
│
└── Password hashing algorithm:
    → Must be bcrypt/argon2, NOT md5/sha256
    → Verify cost factor is adequate (bcrypt rounds ≥ 12)
```

**Application / Domain:**
```
HIGH — JWT Payload & Permission Staleness
├── If roles/permissions are embedded in JWT payload:
│   → Admin changes user role → user's existing token still has 
│     old permissions until expiry
│   → Mitigation: Keep access token short-lived (5-15 min), 
│     re-fetch permissions on refresh
│   → OR: Don't embed permissions in JWT, fetch from DB on each 
│     request (performance trade-off)
│
├── JWT payload should contain at minimum:
│   [sub (user_id), tenant_id, iat, exp, jti (token ID)]
│   → tenant_id is CRITICAL for multi-tenancy
│   → jti enables per-token revocation
│
└── Token expiry configuration:
    ├── Access token: 5-15 minutes (recommended)
    ├── Refresh token: 7-30 days
    └── Verify these are not hardcoded but configurable
```

**Infrastructure — Multi-Tenancy:**
```
CRITICAL — Tenant Isolation
├── Every DB query that returns business data MUST include 
│   WHERE tenant_id = :current_tenant
│   → This should be a global scope / model trait, NOT manual 
│     per-query filtering
│   → One missed filter = cross-tenant data leak
│
├── TenantMiddleware must:
│   1. Extract tenant_id from JWT
│   2. Set it in request context / app container
│   3. Apply global scope to all Eloquent models (if Laravel) 
│      or equivalent
│
├── Admin/superadmin endpoints:
│   → Must use separate route group with explicit guard
│   → Must NOT share the same middleware stack as tenant endpoints
│
└── Verify: Can a user craft a request with a different tenant_id 
    in the body/params and bypass the JWT tenant_id?
    → tenant_id must ALWAYS come from JWT, never from request input
```

**Validation / Auth:**
```
HIGH — Endpoint-Level Authorization
├── Route-group-level auth is NOT sufficient alone
│   → Every controller method should verify specific permission
│   → Example: A "warehouse_worker" role hitting 
│     DELETE /api/shipments/:id should be rejected even if they 
│     have a valid JWT
│
├── Verify authorization granularity:
│   ├── Route group: [authenticated] ← necessary but not sufficient
│   ├── Middleware: [role:admin] ← coarse, better than nothing
│   └── Policy/Gate: [can:delete-shipment] ← ideal granularity
│
└── API endpoints without auth (public):
    → Audit ALL routes — any business endpoint without auth 
      middleware is a P0 vulnerability
```

### Risks
- **P0**: Refresh token not stored server-side → irrevocable sessions
- **P0**: Missing tenant_id global scope → cross-tenant data leakage
- **P1**: No rate limiting → brute force vulnerability
- **P1**: Stale permissions in long-lived JWT → privilege escalation window
- **P2**: No concurrent refresh handling → random logouts for multi-tab users

### Verification
- [ ] Inspect RefreshToken table schema — confirm existence, rotation columns
- [ ] Test: Login → change role via admin → verify old token permissions
- [ ] Test: Send request with forged tenant_id in body — verify rejected
- [ ] Test: 10 rapid failed logins → verify lockout behavior
- [ ] Audit route file — list all endpoints without auth middleware
- [ ] Test: Logout → reuse refresh token → must be rejected

### Changes / Findings

**API:**
```
CRITICAL — Import Idempotency
├── What happens when the same IssOrder (same external_id/reference) 
│   is imported twice?
│   ├── Scenario A: Upsert → existing order updated, existing 
│   │   shipment potentially overwritten → DANGEROUS if shipment 
│   │   is already in Picking state
│   ├── Scenario B: Reject duplicate → return 409 Conflict → SAFE
│   ├── Scenario C: Silently skip → RISKY (user thinks import 
│   │   succeeded with new data)
│   └── Recommended: Reject with 409 if shipment exists beyond Draft,
│       allow re-import (upsert) only if shipment is still Draft
│
├── Idempotency key mechanism:
│   → Import endpoint should accept/generate idempotency key
│   → Duplicate POST with same key within window → return cached result
│   → Prevents duplicate shipments from retry/double-click
│
└── Bulk import atomicity:
    ├── If importing 100 orders, transaction boundary?
    │   ├── All-or-nothing (single transaction) → simple but harsh
    │   ├── Per-order transaction → partial success, need error report
    │   └── Recommended: Per-order transaction + return success/fail 
    │       summary per order
    ├── If per-order: failed orders must NOT create partial data 
    │   (orphan shipment without items)
    └── Import result must include: 
        [order_ref, status(success/fail), error_message, shipment_id]
```

**Application:**
```
CRITICAL — Data Validation on Import
├── Each order line item:
│   ├── SKU must exist in product/inventory master → 
│   │   FK validation or lookup
│   ├── quantity must be > 0 (integer, not float unless unit allows)
│   ├── price >= 0 (zero acceptable for samples/gifts, 
│   │   negative NOT acceptable)
│   ├── Required fields: [sku, quantity, description] → 
│   │   reject entire order if missing
│   └── Max line items per order? (DoS protection — 
│       import with 100K line items)
│
├── Order-level validation:
│   ├── Customer/recipient address validation (at least non-empty)
│   ├── Order date not in the future? (business rule dependent)
│   ├── Currency code valid (ISO 4217)?
│   └── Total amount = sum of line items? (if total is provided)
│
└── SKU validation depth:
    ├── Minimum: SKU exists in product catalog
    ├── Ideal: SKU is active (not discontinued)
    └── Edge case: SKU exists but zero stock → 
        should import succeed? (probably yes, stock check at 
        shipment preparation, not import)
```

**Domain:**
```
HIGH — Order → Shipment Mapping
├── Cardinality:
│   ├── 1 Order = 1 Shipment (simplest, likely current)
│   ├── 1 Order = N Shipments (split shipment — needed for 
│   │   multi-warehouse or partial availability)
│   └── N Orders = 1 Shipment (consolidation — 
│       same customer, same address)
│   → Document current mapping and verify it's enforced
│
├── Shipment creation:
│   ├── Initial status MUST be "Draft" (not skipping to Warehouse)
│   ├── created_by must be set (audit trail)
│   ├── tenant_id must be set from context (not from import data)
│   └── Shipment items must mirror order items 1:1 at creation
│
└── Stock reservation at import time:
    ├── If reservation happens at import → immediate stock lock, 
    │   good for accuracy, bad for flexibility
    ├── If reservation happens at Draft→Warehouse transition → 
    │   better, allows editing in Draft
    └── If no reservation at all → oversell risk, multiple 
        shipments compete for same stock during picking
    → Verify current behavior and document it
```

**Database:**
```
HIGH — Schema Integrity
├── iss_orders table:
│   ├── unique constraint on external_id + tenant_id? 
│   │   (prevent duplicate import at DB level)
│   ├── index on external_id for lookup performance
│   └── FK to shipments (or shipments FK to iss_orders)?
│
├── shipments table:
│   ├── FK to iss_order_id (nullable if manual creation possible)
│   ├── status column: VARCHAR or ENUM? 
│   │   → ENUM is safer for DB-level constraint
│   ├── version column for optimistic locking?
│   └── Composite index on [tenant_id, status] for dashboard queries
│
└── shipment_items table:
    ├── FK to shipment_id (CASCADE DELETE or RESTRICT?)
    │   → RESTRICT is safer: don't delete shipment with items
    ├── FK to sku/product_id
    ├── quantity_ordered vs quantity_picked columns?
    └── CHECK constraint: quantity > 0
```

### Risks
- **P0**: No duplicate import protection → multiple shipments for same order
- **P0**: Partial import failure with no transaction boundary → orphaned data
- **P1**: SKU validation skipped → shipment created for non-existent product → fails at picking
- **P1**: No idempotency → retry creates duplicates
- **P2**: Missing audit trail on import (who imported, when, source file)

- [ ] Test: Import file with 1 valid + 1 invalid order → verify valid one succeeds, invalid rolls back completely

# Architect Agent — Full Audit Review

---

## Change

End-to-end architectural review of the entire application across 5 domains: Auth/JWT, IssOrder Import & Shipment Creation, Shipment Lifecycle State Machine, Warehouse Operations, and Purchasing Module. Evaluating cross-layer consistency, boundary violations, responsibility misplacement, schema integrity, state machine soundness, and long-term systemic risks based on Backend and Frontend agent findings.

---

## Verdict

**Conditional Approval**

The system's overall domain decomposition (Auth, Orders, Shipments, Warehouse, Purchasing) is sound. However, there are **critical architectural gaps** that, if not addressed, will produce data corruption, security breaches, and cascading failures under real-world load. The system can proceed to production only after the Required Conditions below are met.

---

## Reason

### 1 — Auth & Tenant Isolation: Foundational Layer at Risk

```
ARCHITECTURAL CONCERN: Trust Boundary Violation

The entire system's security posture depends on two invariants:
  1. Every request is authenticated (JWT valid + not revoked)
  2. Every data access is tenant-scoped

If either invariant is enforced inconsistently — via manual 
per-query filtering, optional middleware, or implicit convention 
rather than structural guarantee — the architecture has a 
foundational trust boundary violation.

Current risk signals from Backend analysis:
├── Tenant isolation may rely on developer discipline 
│   (per-query WHERE) rather than structural enforcement 
│   (global scope / row-level security)
├── Refresh token lifecycle may be stateless-only 
│   (no server-side record → irrevocable tokens)
├── Permission enforcement may be at route-group level only 
│   (not per-endpoint policy)
└── JWT payload may carry stale permissions with no 
    forced re-validation window

Architectural judgment:
→ Tenant isolation MUST be a structural guarantee, not a convention.
   This is a non-negotiable boundary. Acceptable implementations:
   (a) ORM global scope applied at model base class (all models inherit)
   (b) Database-level Row Level Security (RLS) policy
   (c) Schema-per-tenant isolation (heavy, usually not needed)
   
   Option (a) is the minimum viable architectural safeguard.
   If ANY model can bypass the tenant scope without explicit 
   withoutTenantScope() override, the system is architecturally unsound.

→ Token revocation MUST have server-side state.
   Stateless JWT alone is acceptable for access tokens (short-lived).
   Refresh tokens MUST be stored server-side with revocation capability.
   This is a layer boundary issue: the auth layer must own session 
   lifecycle, not delegate it entirely to the client.
```

### 2 — State Machine: The Central Architectural Element

```
ARCHITECTURAL CONCERN: State Machine Ownership & Enforcement

The shipment state machine (Draft → Warehouse → Picking → Ready 
→ Preparing → Delivered) is the backbone of the entire operational 
flow. It connects Orders, Warehouse, and Purchasing (stock) domains.

Architectural requirements:
├── Single Source of Truth: The allowed-transitions map MUST exist 
│   in exactly ONE place in the codebase. Not duplicated across 
│   controllers, services, frontend constants, and UI components.
│   
│   Correct layering:
│   ┌─────────────────────────────────────────────────────┐
│   │ Domain Layer: ShipmentStateMachine                   │
│   │   → Defines states, transitions, guards              │
│   │   → Single authority for "can X transition to Y?"    │
│   ├─────────────────────────────────────────────────────┤
│   │ Application Layer: ShipmentTransitionService         │
│   │   → Calls state machine for validation               │
│   │   → Orchestrates side effects (stock, notifications) │
│   ├─────────────────────────────────────────────────────┤
│   │ API Layer: ShipmentController                        │
│   │   → Receives transition request                      │
│   │   → Delegates to application layer                   │
│   │   → Returns result                                   │
│   ├─────────────────────────────────────────────────────┤
│   │ Frontend: reads allowed transitions from API         │
│   │   → Does NOT define its own transition rules         │
│   │   → Renders buttons based on API-provided actions    │
│   └─────────────────────────────────────────────────────┘
│   
│   If frontend has its own copy of transition rules → they WILL 
│   drift from backend → users see buttons that don't work, or 
│   don't see buttons for actions they can take.
│
├── Missing States: The current model lacks:
│   ├── Cancelled — essential for any real operation
│   ├── OnHold — essential for exception handling
│   └── PartiallyPicked — if the business needs it
│   
│   These are not feature requests; they are architectural gaps.
│   A state machine without terminal-exception states forces the 
│   system into undefined behavior when real-world exceptions occur.
│   Warehouse worker can't find 3 of 10 items → what state?
│   If no PartiallyPicked or OnHold → system is stuck.
│
├── Concurrency Control: State transitions are a classic 
│   lost-update problem.
│   ├── Optimistic locking (version column) is the minimum 
│   │   viable approach
│   ├── Without it: two users simultaneously transition the same 
│   │   shipment → one write silently overwrites the other
│   ├── DB-level: CHECK constraint or trigger on valid transitions 
│   │   is defense-in-depth (recommended, not required)
│   └── The version field MUST be:
│       (a) Returned in every API response for the shipment
│       (b) Required in every state-transition request
│       (c) Checked in the UPDATE WHERE clause
│
└── Side Effects per Transition: Each transition triggers 
    cross-domain side effects that must be architecturally owned:
    
    ┌───────────────────┬────────────────────────────────────┐
    │ Transition        │ Side Effect (Domain Owner)         │
    ├───────────────────┼────────────────────────────────────┤
    │ Draft→Warehouse   │ Stock reservation (Inventory)      │
    │ Warehouse→Picking │ Pick list generation (Warehouse)   │
    │ Picking→Ready     │ Reservation confirmation (Inv.)    │
    │ Ready→Preparing   │ Label/carrier creation (Shipping)  │
    │ Preparing→Deliver │ Tracking activation (Shipping)     │
    │ Any→Cancelled     │ Stock release (Inventory)          │
    │                   │ Notification (Notification)        │
    │ Any→OnHold        │ Alert + reason logging (Ops)       │
    └───────────────────┴────────────────────────────────────┘
    
    These side effects MUST be:
    (a) Triggered by the application/domain layer, not the controller
    (b) Transactionally consistent OR eventually consistent with 
        explicit compensation (saga/outbox)
    (c) Failure of a side effect must NOT leave the shipment in 
        an inconsistent state (e.g., status changed but stock 
        not reserved)
```

```

┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Orders     │────▶│  Shipments   │◀────│  Warehouse   │
│              │     │              │     │              │

                            │

└── Inventory is the INTEGRATION HUB:
    → It must expose a clean interface:
       reserve(sku, qty, reference_type, reference_id)
       release(reservation_id)
       commit(reservation_id)
       receive(sku, qty, warehouse_id, zone_id, source_type, source_id)
       getAvailable(sku, warehouse_id?) → available_qty
    → If this interface doesn't exist and modules write directly 
      to stock tables → cross-domain coupling will make the 
      system unmaintainable.
```

### 4 — Import Architecture: Transaction Boundary Design

```
ARCHITECTURAL CONCERN: Bulk Operation Atomicity

The IssOrder import is a bulk operation that creates N orders 
and potentially N shipments. The transaction boundary choice 
has cascading architectural implications:

Option A — Single Transaction (all-or-nothing):
├── Pro: Atomic, simple to reason about
├── Con: 1 bad row fails 999 good rows
├── Con: Long transaction → DB lock contention
└── Verdict: Unacceptable for production bulk imports

Option B — Per-Order Transaction:
├── Pro: Partial success, isolated failures
├── Pro: Short transactions, low lock contention
├── Requirement: Each order+shipment creation in its own 
│   transaction boundary
├── Requirement: Import result must report per-order outcome
├── Con: More complex error reporting
└── Verdict: RECOMMENDED

Option C — Async Queue per Order:
├── Pro: Non-blocking, scalable
├── Pro: Automatic retry on transient failures
├── Con: UI must handle eventual consistency (polling/WS)
├── Con: Import result not immediately available
└── Verdict: Best for large-scale (>1000 orders), 
    overkill for small imports

Architectural decision needed:
→ For imports ≤ 500 orders: Option B (sync, per-order tx)
→ For imports > 500 orders: Option C (async, queued)
→ MUST NOT be Option A in any case

Additionally:
├── Idempotency key per order (external_id + tenant_id) 
│   MUST have DB unique constraint
├── Import batch should be tracked as a first-class entity:
│   [import_id, file_name, user_id, tenant_id, started_at, 
│    completed_at, total_rows, success_count, fail_count, status]
└── This enables import history, retry, and audit trail
```

### 5 — Purchasing → Inventory: Receive Flow Architecture

```
ARCHITECTURAL CONCERN: GoodsReceipt Impact Radius

GoodsReceipt is the entry point for physical stock into the system.
It has cross-cutting impact on:
├── Inventory levels (direct write)
├── Pending shipment reservations (may auto-fulfill)
├── Purchase Order status (partial → complete)
└── Potentially: replenishment alerts, zone capacity

Architecture requirements:
├── GoodsReceipt MUST be an auditable, immutable event:
│   ├── Once created, a GR cannot be edited — only reversed 
│   │   with a new compensating GR (negative/reversal)
│   ├── This is an accounting-grade requirement for stock integrity
│   └── If GRs are editable → stock audit trail is unreliable
│
├── Over-receive policy MUST be configurable:
│   ├── Hard block: reject if qty > PO qty remaining
│   ├── Tolerance: accept if within X% (configurable per tenant)
│   └── Approval-required: flag for manager approval if over
│   → This is a BUSINESS RULE that must live in domain layer, 
│     not in controller validation
│
├── Receive → Stock → Reservation cascade:
│   ├── When new stock is received, should the system 
│   │   auto-match against pending reservations?
│   ├── If yes: this is an event-driven process 
│   │   (GoodsReceived event → ReservationMatcher listener)
│   ├── If no: manual trigger or batch job
│   ├── Auto-match has fairness implications: 
│   │   which shipment gets the stock? FIFO by creation date? 
│   │   Priority? → Business rule needed.
│   └── This cross-domain interaction (Purchasing → Inventory 
│       → Shipments) is the most architecturally complex flow 
│       in the system and must be explicitly designed.
│
└── GR reversal impact:
    ├── If stock from this GR was already picked for a shipment → 
    │   reversal would make stock negative
    ├── System MUST check: is this stock committed? 
    │   If yes → block reversal OR force shipment cancellation
    └── This is a distributed constraint that requires careful 
        orchestration
```

P0-2: REFRESH TOKEN SUNUCU TARAFINDA SAKLANMIYOR/İPTAL EDİLEMİYOR
──────────────────────────────────────────────────────────────────
Durum: Backend analizinde refresh token'ın DB'de saklanıp

```
P0-3: STATE MACHINE'DE CONCURRENCY KONTROLÜ YOK
─────────────────────────────────────────────────
Durum: Shipment state transition'larında optimistic locking 
(version column) veya herhangi bir concurrency mekanizması 
doğrulanamamış.

Etki:
  - İki warehouse çalışanı aynı anda aynı shipment'ı 
    "Picking → Ready" yapar → ikisi de başarılı olur → 
    audit trail bozulur, side effect'ler iki kez çalışır
  - Stok double-commit riski: aynı reservation iki kez 
    onaylanır → stok negatife düşer
  - Picking sırasında eş zamanlı cancel → cancel işlemi 
    kaybolur, shipment "Ready" olarak devam eder

Gerekli aksiyon:
  1. shipments tablosuna `version` (integer, default 0) kolonu ekle
  2. Her state transition: 
     UPDATE shipments SET status = :new, version = version + 1 
     WHERE id = :id AND version = :expected_version
  3. 0 rows affected → 409 Conflict döndür
  4. API response'da version alanı her zaman dön
  5. Frontend 409'u handle etsin: "Bu kayıt başkası tarafından 
     değiştirildi. Yenileyin."

Kabul kriteri:
  - İki eş zamanlı transition request → sadece biri başarılı, 
    diğeri 409 Conflict
  - version alanı her shipment response'unda mevcut
