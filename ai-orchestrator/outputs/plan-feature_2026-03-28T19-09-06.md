#### 2. Başlangıç Noktası Resolve (StartLocationResolver)

```
Girdi: RouteOptimizationRequestDto
Çıktı: ResolvedStartPoint { Latitude, Longitude, Address }
```

```
Pseudo-logic:

EĞER StartLocationType == null → eski davranış: StartAddress'ten geocode (mevcut akış)
EĞER StartLocationType == CurrentLocation:
    KONTROL: StartLatitude && StartLongitude dolu olmalı
    DÖNÜŞ: { StartLatitude, StartLongitude, Address = reverse geocode veya null }
    
EĞER StartLocationType == Warehouse:
    KONTROL: StartWarehouseId dolu olmalı
    Warehouse = _warehouseRepository.GetById(StartWarehouseId)
    KONTROL: Warehouse != null (yoksa → NotFoundException)
    DÖNÜŞ: { Warehouse.Latitude, Warehouse.Longitude, Warehouse.Address }
    
EĞER StartLocationType == ManualAddress:
    KONTROL: StartAddress dolu olmalı
    EĞER StartLatitude && StartLongitude dolu → direkt kullan (frontend geocode yapmışsa)
    DEĞİLSE → Geocoding API ile lat/lng çöz
    DÖNÜŞ: { lat, lng, StartAddress }
```

#### 3. Sıralama Pipeline (RouteOrderingService)

```
Girdi: 
  - ResolvedStartPoint (lat, lng)
  - List<ProjectStop> (Code, Lat, Lng, DeliveryWindowStart?, DeliveryWindowEnd?)
  - VehicleType
  - ReturnToStart

Çıktı:
  - List<OrderedWaypoint> (sıralı waypoint listesi, köprü waypoint'leri dahil)
```

```
PIPELINE:

─────────────────────────────────────
ADIM 1: Yaka Ayırma
─────────────────────────────────────
başlangıçYaka = StartLng >= 29.05 ? Anadolu : Avrupa
avrupaGrubu = stops.Where(s => s.Longitude < 29.05)
anadoluGrubu = stops.Where(s => s.Longitude >= 29.05)

başlangıçYakaGrubu = başlangıçYaka == Avrupa ? avrupaGrubu : anadoluGrubu
diğerYakaGrubu = başlangıçYaka == Avrupa ? anadoluGrubu : avrupaGrubu

─────────────────────────────────────
ADIM 2: Time Window Gruplama (her yaka grubu için ayrı ayrı)
─────────────────────────────────────
TimeWindowBucket sıralama fonksiyonu:

  GetBucket(stop):
    EĞER stop.DeliveryWindowStart == null → BUCKET_NONE (3. öncelik)
    EĞER stop.DeliveryWindowEnd <= 12:00 → BUCKET_MORNING (1. öncelik)
    EĞER stop.DeliveryWindowStart >= 12:00 → BUCKET_AFTERNOON (2. öncelik)
    ELSE → BUCKET_MORNING (12'yi kapsayan pencereler sabah sayılır)

  Her yaka grubu içinde:
    sabahGrubu = stops.Where(bucket == MORNING)
    öğledenSonraGrubu = stops.Where(bucket == AFTERNOON)
    penceresizGrubu = stops.Where(bucket == NONE)

─────────────────────────────────────
ADIM 3: Mesafe Sıralama (her bucket içinde)
─────────────────────────────────────
Her bucket içinde:
  → Haversine ile başlangıç noktasına olan mesafeyi hesapla
  → EN UZAKTAN EN YAKINA sırala (descending distance)

─────────────────────────────────────
ADIM 4: Birleştirme
─────────────────────────────────────
başlangıçYakaSıralı = [sabahGrubu_mesafeSıralı] + [öğledenSonraGrubu_mesafeSıralı] + [penceresizGrubu_mesafeSıralı]
diğerYakaSıralı = [sabahGrubu_mesafeSıralı] + [öğledenSonraGrubu_mesafeSıralı] + [penceresizGrubu_mesafeSıralı]

─────────────────────────────────────
ADIM 5: Köprü Ekleme
─────────────────────────────────────
EĞER diğerYakaGrubu.Count > 0:
    köprüWaypoint = VehicleType == Kamyon ? YSS_KOORDINATLARI : FSM_KOORDINATLARI
    finalSıra = başlangıçYakaSıralı + [köprüWaypoint] + diğerYakaSıralı
EĞER diğerYakaGrubu.Count == 0:
    finalSıra = başlangıçYakaSıralı   // Köprü eklenmez

─────────────────────────────────────
ADIM 6: ReturnToStart
─────────────────────────────────────
EĞER ReturnToStart == true:
    EĞER son durak farklı yakadaysa:
        finalSıra += [köprüWaypoint]   // Geri dönüş köprüsü
    finalSıra += [StartPoint]          // Başlangıç noktası son waypoint olarak

DÖNÜŞ: finalSıra
```

#### 4. Haversine Helper

```csharp
public static class HaversineHelper
{
    /// <summary>
    /// İki koordinat arası kuş uçuşu mesafe (km)
    /// </summary>
    public static double CalculateDistanceInKm(
        double lat1, double lon1, 
        double lat2, double lon2)
    {
        const double R = 6371; // Dünya yarıçapı km
        var dLat = ToRad(lat2 - lat1);
        var dLon = ToRad(lon2 - lon1);
        var a = Math.Sin(dLat / 2) * Math.Sin(dLat / 2) +
                Math.Cos(ToRad(lat1)) * Math.Cos(ToRad(lat2)) *
                Math.Sin(dLon / 2) * Math.Sin(dLon / 2);
        var c = 2 * Math.Atan2(Math.Sqrt(a), Math.Sqrt(1 - a));
        return R * c;
    }

    private static double ToRad(double deg) => deg * Math.PI / 180;
}
```

#### 5. Time Window Uyarı Üretimi

```
Rota sıralama sonrası, Google Routes API'den dönen leg sürelerine göre:

tahminiSaat = başlangıçSaati  // Parametrik veya default 08:00
FOR EACH waypoint IN finalSıra:
    tahminiSaat += öncekiLeg.duration  // Google Routes API'den gelen süre
    EĞER waypoint.DeliveryWindowStart != null:
        EĞER tahminiSaat < waypoint.DeliveryWindowStart VEYA tahminiSaat > waypoint.DeliveryWindowEnd:
            warnings.Add(new TimeWindowWarningDto {
                ProjectCode = waypoint.ProjectCode,
                WindowStart = waypoint.DeliveryWindowStart.ToString("HH:mm"),
                WindowEnd = waypoint.DeliveryWindowEnd.ToString("HH:mm"),
                EstimatedArrival = tahminiSaat.ToString("HH:mm"),
                Message = $"{waypoint.ProjectCode} projesi için teslimat penceresi " +
                          $"{windowStart}-{windowEnd}, tahmini varış: {estimated}"
            })
```

> **Varsayım:** Başlangıç saati parametrik olarak 08:00 kabul edilir veya request'e `DepartureTime` alanı eklenebilir. İlk fazda 08:00 default yeterli. İleride DepartureTime alanı eklenebilir.

#### 6. Ana Orchestration (RouteOptimizationService)

```
public async Task<RouteOptimizationResponseDto> OptimizeRoute(RouteOptimizationRequestDto request)
{
    // 1. Başlangıç noktası çöz
    EĞER request.StartLocationType != null:
        startPoint = await _startLocationResolver.Resolve(request)
    ELSE:
        startPoint = mevcutGeocodingAkışı(request.StartAddress)  // backward compat

    // 2. Proje verilerini çek (DeliveryWindow dahil)
    projects = await _projectRepository.GetByCodesWithDeliveryWindow(request.ProjectCodes)
    stops = projects.Select(p => new ProjectStop(p.Code, p.Lat, p.Lng, p.DeliveryWindowStart, p.DeliveryWindowEnd))

    // 3. Sıralama pipeline
    EĞER request.StartLocationType != null:  // Yeni akış
        orderedWaypoints = _routeOrderingService.OrderStops(startPoint, stops, request.VehicleType, request.ReturnToStart)
    ELSE:
        orderedWaypoints = null  // Eski akış: Google optimize etsin

    // 4. Google Routes API çağrısı
    EĞER orderedWaypoints != null:
        googleResponse = await _googleRoutesApi.ComputeRoutes(
            origin: startPoint,
            waypoints: orderedWaypoints,   // Sıralı
            destination: request.ReturnToStart ? startPoint : orderedWaypoints.Last(),
            optimizeWaypointOrder: false    // ← ARTIK FALSE
        )
    ELSE:
        googleResponse = await mevcutGoogleÇağrısı()  // optimizeWaypointOrder: true (eski davranış)

    // 5. Time window uyarıları üret
    warnings = _routeOrderingService.GenerateTimeWindowWarnings(orderedWaypoints, googleResponse.Legs)

    // 6. Response oluştur
    return new RouteOptimizationResponseDto {
        // ... mevcut alanlar
        TimeWindowWarnings = warnings
    }
}
```

## User Flow

### Flow 1 — Başlangıç Noktası Seçimi
1. **Entry:** Kullanıcı Rota Optimizasyon sayfasını açar
2. **Action:** Mevcut `StartAddress` text input yerine 3 seçenekli radio group görür:
   - **"Mevcut Konum"** seçerse → tarayıcı `navigator.geolocation.getCurrentPosition()` tetiklenir → konum alınırsa lat/lng state'e yazılır, adres bilgisi gösterilir → konum reddedilirse inline hata mesajı + otomatik "Manuel Adres" seçeneğine fallback önerisi
   - **"Depo Seçimi"** seçerse → Warehouse dropdown görünür (API'den liste çekilir) → seçim yapar → seçilen deponun adı/adresi görüntülenir
   - **"Manuel Adres"** seçerse → text input alanı gösterilir (mevcut StartAddress davranışı)
   - Altında **"Başlangıca Geri Dön"** checkbox'ı → default unchecked
3. **Result:** Form submit edildiğinde DTO'ya `StartLocationType`, ilgili alanlar ve `ReturnToStart` gönderilir

## Critical Issues

### 1. Sıralama Algoritması Yön Hatası — Mantık Tutarsızlığı
Backend analizinde sıralama "en uzaktan en yakına (descending distance)" olarak tanımlanmış. Ancak bu, **rota sırasında ilk gidilecek durak en uzak nokta** demek. Bir yaka içinde en uzak noktaya gidip sonra geri dönmek, arada zikzak yaratır. Bu greedy nearest-neighbor değil, greedy **farthest-first** yaklaşımı. Çalışır ama netleştirilmesi gereken nokta şu: her bucket içinde duraklar birbirine göre mi, yoksa başlangıç noktasına göre mi sıralanıyor? Backend analizi "başlangıç noktasına göre" diyor ama bu durumda A→(en uzak)→(2. en uzak)→...→(en yakın) rotası, duraklar arasında geriye atlamalar yaratabilir. **Bucket içi sıralamada duraklar arası mesafe (chain/sequential nearest-neighbor) düşünülmeli veya mevcut yaklaşımın bilinçli bir trade-off olduğu açıkça belirtilmeli.**

### 2. DepartureTime Eksikliği — Uyarı Doğruluğunu Doğrudan Etkiler
Tahmini varış saati hesaplaması 08:00 hardcoded başlangıçla yapılıyor. Kullanıcı 14:00'te rota oluşturursa tüm time window uyarıları yanlış olacak. Bu "ilk fazda kabul edilir" denmiş ancak **uyarının yanlış olması, uyarı olmamasından daha kötüdür** — kullanıcı güvenini sarsar. En azından request'e `departureTime` opsiyonel alanı **ilk release'de** eklenmeli, default 08:00 korunabilir.

### 3. Durak Süresi (Dwell/Service Time) Hesaba Katılmamış
Tahmini varış hesaplamasında leg süreleri toplanıyor ama her duraktaki teslimat/boşaltma süresi (dwell time) yok. 10 duraklı rotada her durakta 15 dk bile olsa 2.5 saat kayıp oluşur. Time window uyarılarının neredeyse tamamı yanlış çıkabilir. **En azından sabit bir default dwell time (ör: 15 dk) eklenmeli.**

### 4. Warehouse Lat/Lng Varsayımı Doğrulanmamış
`StartLocationResolver`'da Warehouse seçildiğinde `Warehouse.Latitude` ve `Warehouse.Longitude` okunuyor. Ancak mevcut Warehouse entity'de bu alanların varlığı **varsayılmış**, doğrulanmamış. Eğer yoksa tüm Warehouse akışı çöker. Architect de bu riske değinmiş ama çözüm belirsiz bırakılmış.

10. **Aynı koordinatlarda iki farklı proje** → Haversine mesafe 0, sıralama kararsız (stable sort gerekli)
11. **Başlangıç noktası durak koordinatıyla aynı** → mesafe 0, sorun çıkmamalı
12. **Google Routes API timeout/hata** → uyarı üretimi yapılamaz, partial response dönmemeli
13. **Çok sayıda durak (25+ waypoint)** → Google Routes API waypoint limiti aşılır mı?

## Edge Cases

- **Longitude 29.05 sınır değeri:** Backend'de `>=` kullanılmış ama bu Boğaz'ın ortasına denk gelir. Boğaz üzerinde proje olma ihtimali düşük ama köprü koordinatları bu eşiğe yakın — köprü waypoint'inin kendisi hangi yakaya düşüyor?
- **FSM Köprüsü koordinatı (~29.05-29.06 arasında):** Köprü waypoint'i "diğer yaka" kontrol mantığında yaka tespitine dahil edilmemeli, sadece transit waypoint olarak eklenmeli
- **Aynı anda iki kullanıcı aynı depo seçip farklı rotalar oluşturursa:** Sorun yok — read-only Warehouse, concurrency sorunu yok
- **TimeOnly midnight cross:** DeliveryWindow 23:00-01:00 gibi gece yarısını geçen pencere → Start > End validation'a takılır. Bu senaryonun out of scope olduğu belirtilmeli
- **Geolocation accuracy:** Düşük accuracy koordinat (şehir düzeyi) gelirse Haversine hesaplama hâlâ anlamlı mı? Yaka tespiti yanlış olabilir

## Missing Checks

1. **Warehouse entity'de Latitude/Longitude alanlarının varlığı ve nullability kontrolü** — Warehouse seçilip lat/lng null gelirse? Validation'da `Warehouse.Latitude != 0 && Warehouse.Longitude != 0` kontrolü gerekli
2. **Google Routes API waypoint limiti** — Google Routes API'de maximum waypoint sayısı 25 (origin + destination + 23 intermediate). Backend'de bu limit kontrol edilmiyor. Yaka sıralaması + köprü waypoint'leri bu sayıyı artırır
3. **Köprü waypoint'inin ReturnToStart senaryosunda çift eklenmesi** — Pipeline'da gidiş köprüsü + dönüş köprüsü ayrı ayrı ele alınmalı, aynı köprü mü farklı köprü mü?
4. **VehicleType ile köprü seçimi — yeni pipeline'da doğru aktarılıyor mu?** — `RouteOrderingService`'e `VehicleType` geçirilmeli
5. **Frontend'de radio button default değeri** — `ManualAddress` default, ama sayfa ilk açıldığında StartAddress boş. Submit validation tetiklenmeli
6. **Time window uyarısı: erken varış durumu** — Backend analizi "pencere dışı" diyor ama erken varış (tahmini 08:30, pencere 10:00-12:00) da uyarı üretmeli mi? Lojistik açıdan erken varış sorun olmayabilir. **İş kuralı netleştirilmeli**
7. **Response'ta time window warning'lerinin sırası** — Rota sırasına göre mi, proje koduna göre mi? Kullanıcı deneyimi açısından rota sırasına göre olmalı
8. **Frontend: Depo listesi boşsa (hiç Warehouse tanımlı değilse)** — Dropdown boş, kullanıcı seçemez. UX'te "Tanımlı depo bulunmamaktadır" mesajı gösterilmeli
9. **Concurrent request'lerde Geolocation cache** — `maximumAge: 300000` (5 dk) ayarlandı ama kullanıcı konumunu değiştirmişse eski koordinat gidebilir. Rota optimizasyonu sayfasında "Konumu yenile" butonu olmalı

## Simplifications

1. **TimeWindowWarningDto'da `Message` alanı gereksiz olabilir** — `ProjectCode`, `WindowStart`, `WindowEnd`, `EstimatedArrival` zaten mevcut. Message frontend'de formatlanabilir. Backend'in UI mesajı üretmesi separation of concerns ihlali. **Öneri:** Backend'den Message kaldırılsın, frontend kendi locale'ine göre formatlasın. Ancak bu minor — blocking değil.

2. **Bucket sınıflandırması overengineering olabilir** — Mevcut gereksinim "sabah pencereli önce, öğleden sonra sonra" diyor. Basitçe `DeliveryWindowStart` ascending sıralama aynı sonucu verir:
   - 08:00 başlangıçlı pencere → öne gelir
   - 14:00 başlangıçlı pencere → arkaya gider
   - null pencere → en sona (COALESCE ile TimeOnly.MaxValue)
   
   Bu, bucket mantığı + 12:00 boundary edge case sorununu tamamen ortadan kaldırır. **Kuvvetle önerilir.**

3. **`geolocationStatus` üç durumlu ('idle' | 'loading' | 'success' | 'error') — dört durumlu** — Naming minor ama 'idle' gereksiz, component mount'ta direkt 'loading' tetiklenmiyorsa anlamlı.

## Recommendation

### Mutlaka Yapılması Gerekenler (Blocking)

1. **Warehouse entity'de Latitude/Longitude varlığını doğrula.** Yoksa Warehouse akışını Geocoding fallback'i ile destekle veya akışı devre dışı bırak. Bu doğrulanmadan Warehouse seçeneği implement edilmemeli.

2. **Google Routes API waypoint limit kontrolü ekle.** 25 waypoint sınırını aşan request'ler için erken hata dön. Köprü waypoint'leri dahil sayılmalı.

3. **DepartureTime parametresini ilk release'e dahil et** (opsiyonel, default 08:00). Frontend'de basit bir time picker yeterli. Yoksa time window uyarıları güvenilmez olacak ve kullanıcı güveni ilk günden sarsılır.

4. **Bucket sınıflandırması yerine `DeliveryWindowStart` ascending sıralama kullan.** 12:00 boundary edge case'ini ortadan kaldırır, kodu sadeleştirir.

5. **Sıralama algoritmasında chain nearest-neighbor veya en azından mevcut farthest-first yaklaşımının trade-off'unu dökümante et.** Eğer bilinçli bir karar ise kabul edilir, ama "en uz
