**SetZoneDriverInfoCommandHandler — Yeni Orchestration Akışı:**
```csharp
public async Task<Result> Handle(SetZoneDriverInfoCommand request, ...)
{
    var departureTime = request.DepartureTime ?? new TimeOnly(8, 0);
    
    // 1. Mevcut pre-flight kontrolleri (DEĞİŞMEZ)
    var zone = await GetZonePreparation(request.ZonePreparationId);
    var vehicle = await GetVehicle(request.VehicleId);
    // ... mevcut validasyonlar ...

    // 2. Rota optimizasyonu (NON-BLOCKING)
    try
    {
        var projectCoordinates = await GetProjectCoordinatesForZone(zone);
        
        if (projectCoordinates.Count > 1) // Tek proje ise skip
        {
            var optimizedOrder = await _routeOrderingService.OptimizeOrder(
                projectCoordinates,
                startLat: _systemSettings.DepotLatitude,
                startLng: _systemSettings.DepotLongitude,
                vehicleType: vehicle.VehicleType,
                departureTime: departureTime,
                returnToStart: true
            );
            
            // Batch update — transaction içinde
            await _mediator.Send(new BatchUpdateDeliveryOrderCommand(optimizedOrder));
        }
    }
    catch (Exception ex)
    {
        _logger.LogWarning(ex, 
            "Rota optimizasyonu başarısız. ZonePreparationId={Id}. Manuel sıralama korunuyor.", 
            request.ZonePreparationId);
        // Akış devam ediyor — non-blocking
    }

    // 3. Çoklu şoför kaydetme
    var primaryDriverId = request.DriverIds.First();
    
    // Mevcut ZonePreparation.DriverId güncelle (backward compat)
    zone.DriverId = primaryDriverId;
    
    // Join tablosu — önce mevcut kayıtları temizle, sonra yeniden ekle
    await ClearExistingDrivers(zone.Id);
    var driverEntries = request.DriverIds.Select((id, index) => 
        new ZonePreparationDriver
        {
            ZonePreparationId = zone.Id,
            DriverId = id,
            IsPrimary = index == 0
        });
    await _context.ZonePreparationDrivers.AddRangeAsync(driverEntries);

    // 4. Mevcut atama akışı (DEĞİŞMEZ)
    zone.VehicleId = request.VehicleId;
    // ... sevkiyatları AssignedToVehicle statüsüne geçir ...
    // Shipment.DriverId = primaryDriverId (mevcut davranış korunur)
    
    await _context.SaveChangesAsync();
    
    return Result.Success();
}
```

**Validator Güncelleme:**
```csharp
public class SetZoneDriverInfoCommandValidator : AbstractValidator<SetZoneDriverInfoCommand>
{
    public SetZoneDriverInfoCommandValidator()
    {
        RuleFor(x => x.DriverIds)
            .NotEmpty().WithMessage("En az bir şoför seçilmelidir.");
        
        RuleFor(x => x.DriverIds)
            .Must(ids => ids.Distinct().Count() == ids.Count)
            .WithMessage("Aynı şoför birden fazla seçilemez.");
        
        RuleFor(x => x.VehicleId)
            .GreaterThan(0);
        
        RuleFor(x => x.DepartureTime)
            .Must(t => t == null || (t.Value.Hour >= 0 && t.Value.Hour <= 23))
            .WithMessage("Geçerli bir saat giriniz.");
    }
}
```

**BatchUpdateDeliveryOrderCommandHandler:**
```csharp
public async Task<Result> Handle(BatchUpdateDeliveryOrderCommand request, ...)
{
    // Transaction scope — ya hepsi ya hiçbiri
    using var transaction = await _context.Database.BeginTransactionAsync();
    try
    {
        foreach (var item in request.Items)
        {
            var project = await _context.Projects.FindAsync(item.ProjectId);
            if (project == null) 
                throw new NotFoundException($"Project {item.ProjectId} not found");
            project.DeliveryOrder = item.DeliveryOrder;
        }
        
        await _context.SaveChangesAsync();
        await transaction.CommitAsync();
        return Result.Success();
    }
    catch
    {
        await transaction.RollbackAsync();
        throw;
    }
}
```

**Şoför Ekranı Query Güncelleme:**
- Mevcut query: `WHERE Shipment.DriverId = @currentDriverId`
- Ek query (UNION veya OR): `WHERE ZonePreparationDrivers.DriverId = @currentDriverId`
- Bu sayede primary olmayan şoförler de atandıkları rotayı görebilir
- **Assumption**: Şoför ekranı ZonePreparation bazlı değil Shipment bazlı çalışıyor. Bu durumda Shipment.DriverId hâlâ sadece primary driver'ı taşır. Eğer secondary şoförler de shipment seviyesinde görmeli ise, ayrı bir query/view gerekir — **Architect kararı beklenmeli**.

        builder.HasKey(x => x.Id);

## Reason

Genel mimari yaklaşım sağlam. 4 geliştirme birbiriyle uyumlu, katman sınırları korunuyor, mevcut akışa aşırı müdahale yok. Ancak aşağıdaki 4 konuda netleştirme / düzeltme gerekiyor.

### 1. Orchestration Transaction Boundary — Kritik

Backend analizinde rota optimizasyonu (Adım 2) ve şoför/araç ataması (Adım 3) ayrı adımlar olarak tanımlanmış, ancak **transaction scope net değil**.

**Sorun:** Handler'da 3 farklı yazma işlemi var:
- Batch DeliveryOrder update (optimizasyon sonucu)
- ZonePreparationDrivers insert
- ZonePreparation + Shipment status güncelleme

`BatchUpdateDeliveryOrderCommand` kendi transaction'ını açıyor, ardından handler `SaveChangesAsync()` çağırıyor. Bu **nested transaction riski** yaratır.

**Gerekli koşul:** `BatchUpdateDeliveryOrderCommand`'ı ayrı bir MediatR command olarak iç çağrı yapmak yerine, **aynı handler içinde doğrudan DbContext üzerinden batch update yapılmalı**. Tüm yazma işlemleri tek `SaveChangesAsync()` çağrısıyla commit edilmeli. Eğer optimizasyon non-blocking ise (hata durumunda skip), o zaman:

```
Transaction scope:
  ├── [TRY] DeliveryOrder batch update (optimizasyon sonucu)
  ├── ZonePreparationDrivers insert  
  ├── ZonePreparation.DriverId + VehicleId update
  └── Shipment status updates
  → Tek SaveChangesAsync()
```

Optimizasyon **hesaplama** kısmı (RouteOrderingService.OptimizeOrder çağrısı) transaction dışında kalmalı — sadece DB yazma işlemleri tek transaction'da.

`BatchUpdateDeliveryOrderCommand` endpoint'i ayrıca da expose edilebilir (gelecekte manuel kullanım için), ama orchestration akışı içinde **MediatR iç çağrısı olarak kullanılmamalı**.

### 2. ZonePreparation.DriverId Backward Compatibility Stratejisi — Doğru Ama Belgelenmeli

Backend analizi `ZonePreparation.DriverId`'yi koruyup primary driver'ı yansıtma kararı vermiş. Bu doğru yaklaşım. Ancak:

**Gerekli koşul:** Bu field artık **derived/denormalized** bir field. Şu kural kesin olarak belgelenmeli ve handler'da enforce edilmeli:
- `ZonePreparation.DriverId` = `ZonePreparationDrivers` tablosunda `IsPrimary = true` olan kaydın `DriverId`'si
- Bu iki kaynak hiçbir zaman çelişmemeli
- Mevcut codebase'de `ZonePreparation.DriverId`'yi doğrudan yazan başka handler varsa, bu handler'lar da join tablosunu güncellemeli — ya da migration sonrası bir süre için sadece join tablosundan okunacak şekilde read path değişmeli

**Risk:** Eğer mevcut codebase'de `ZonePreparation.DriverId`'yi doğrudan set eden başka bir flow varsa (örn. başka bir command, admin düzeltme), join tablosu ile desync olur.

### 3. Şoför Ekranı Visibility — Architectural Karar Gerekli

Backend analizi doğru bir assumption belirtmiş: Şoför ekranı Shipment bazlı çalışıyor, `Shipment.DriverId` sadece primary driver'ı taşıyor. Secondary şoförlerin rotayı görmesi için:

**Karar:** İki seçenek var:

- **Seçenek A (Önerilen):** Şoför ekranı query'sine `OR EXISTS (SELECT 1 FROM ZonePreparationDrivers zpd WHERE zpd.DriverId = @currentDriverId AND zpd.ZonePreparationId = Shipment.ZonePreparationId)` eklenir. Bu, mevcut Shipment bazlı query'yi genişletir ama Shipment entity'sini kirletmez.

- **Seçenek B:** Her secondary şoför için de Shipment seviyesinde bir kayıt oluşturulur. Bu gereksiz karmaşıklık ve data duplication — **reddetilmeli**.

**Gerekli koşul:** Seçenek A ile ilerlensin. Şoför ekranı query'si ZonePreparationDrivers join'i ile genişletilsin, ancak **Shipment entity'sine yeni FK eklenmemelidir**.

### 4. VehicleType Enum Placement — Domain Layer Doğru

`Domain/Enums/VehicleType.cs` yerleşimi doğru. Vehicle bir domain entity, VehicleType onun intrinsic özelliği. Shared kernel'a taşımaya gerek yok çünkü bu enum sadece Vehicle bounded context'i içinde kullanılıyor. RouteOrderingService'e VehicleType parametresi geçiriliyor ama bu service zaten domain/application katmanında.

**Onay:** Mevcut placement doğru, değişiklik gerekmez.

- 5 projeli bölge, optimizasyon başarılı → DeliveryOrder 1-5 arası güncelleniyor, sıralama optimal
- DepartureTime=09:30 seçili → backend'e "09:30" gidiyor, optimizasyon bu saate göre hesaplanıyor
- Kaydet sonrası shipment'lar AssignedToVehicle statüsüne geçiyor, şoför ekranına düşüyor

**2. Negative:**
- RouteOrderingService.OptimizeOrder timeout/exception → warning log, DeliveryOrder güncellenmez (mevcut sıralama korunur), atama akışı devam eder, frontend warning toast gösterir
- Proje koordinatları eksik/null → OptimizeOrder exception fırlatır → non-blocking catch, devam
- Backend 500 hatası (optimizasyon dışı bir sebep) → frontend error toast, modal açık kalır

**3. Edge:**
- Bölgede **tek proje** → optimizasyon skip edilir (`projectCoordinates.Count > 1` kontrolü), gereksiz API call yapılmaz
- Bölgede **0 proje** → bu durum mümkün mü? Handler'da kontrol var mı? Yoksa guard eklenmeli
- DepartureTime boş/null → backend default 08:00 enforce (`request.DepartureTime ?? new TimeOnly(8, 0)`)
- DepartureTime geçersiz format frontend'den gelirse → backend TimeOnly parse hatası → 400
- Optimizasyon başarılı ama **atama fail** ederse → tek transaction'da olduğu için (Architect Condition #1) her şey rollback → temiz state
- Kullanıcı daha önce **manuel DeliveryOrder** düzenlemiş → otomatik optimizasyon üzerine yazar → pre-flight summary'de "Rota otomatik optimize edilecek" bilgisi gösterilmeli (frontend analizinde eksik)

## Edge Cases

1. **Concurrent atama:** İki operatör aynı bölgeye aynı anda araç atamaya çalışırsa → optimistic concurrency? Mevcut davranış nedir? Belgelenmemiş.

2. **Şoför başka bölgeye zaten atanmış:** DriverIds içindeki bir şoför başka aktif bölgede primary/secondary olarak atanmışsa → bu engellenmeli mi? Business rule eksik.

3. **Araç başka bölgeye zaten atanmış:** Mevcut pre-flight summary'de kontrol var mı? Varsa değişmez, yoksa eksik.

4. **`ZonePreparationDrivers` clear + re-insert race condition:** Handler `ClearExistingDrivers` → `AddRangeAsync` arasında başka request gelirse → tek `SaveChangesAsync` ile atomik olduğu sürece sorun yok, ama concurrent request'lerde unique constraint violation olabilir.

5. **VehicleType enum genişlemesi:** Gelecekte yeni araç tipi eklenirse frontend label mapping güncellenmeli. Backend'de int olarak saklandığı için migration gerekmez ama frontend'de bilinmeyen enum value'ya fallback ("Bilinmeyen") gerekli.

6. **Sekme data staleness:** Client-side filter kullanılıyor, API re-fetch yok. Kullanıcı sayfayı 30 dakika açık tutup sekme değiştirirse stale data görür. Kabul edilebilir ama belgelenmeli.

## Regression Risks

| Alan | Risk | Detay |
|---|---|---|
| Şoför ekranı query | **Orta** | Query ZonePreparationDrivers join'i ile genişletiliyor. Mevcut `WHERE Shipment.DriverId = @id` query'si kırılmamalı — OR/UNION eklenmeli, mevcut koşul korunmalı. |
| Mevcut toggle kullanımı | **Düşük** | Toggle kaldırılıyor, başka component/sayfa bu toggle component'ını import ediyorsa kırılır. Grep ile kontrol gerekli. |
| SetZoneDriverInfoCommand payload | **Yüksek** | `DriverId: int` → `DriverIds: List<int>` breaking change. Bu command'ı çağıran tüm frontend ve test code'ları güncellenmeli. |
| Vehicle DTO | **Düşük** | Yeni alanlar ekleniyor (additive), mevcut consumer'lar kırılmaz. |
| Shipment.DriverId ataması | **Düşük** | Primary driver atanmaya devam ediyor, mevcut davranış korunuyor. |
| Pre-flight summary | **Düşük** | Değişiklik yok ama rota optimizasyon bilgisi eklenmeli (eksik). |

## Missing Checks

1. **DriverIds existence validation:** Backend validator'da `DriverIds` elemanlarının DB'de mevcut olduğu kontrol edilmiyor. Validator'da sadece `NotEmpty` ve `Distinct` var. Handler'da `foreach DriverId → check exists` gerekli, yoksa FK violation.

2. **VehicleId existence check:** Handler'da `GetVehicle(request.VehicleId)` çağrılıyor ama null check açıkça belirtilmemiş. Mevcut davranışta varsa OK, yoksa guard eklenmeli.

3. **Bölgede 0 proje senaryosu:** Handler'da `projectCoordinates.Count > 1` kontrolü var ama `Count == 0` durumu ayrıca guard edilmemiş.

4. **DepartureTime frontend format → backend parse uyumu:** Frontend `"HH:mm"` string gönderiyor, backend `TimeOnly?` bekliyor. Model binding'de `TimeOnly` parse formatı doğru mu? ASP.NET Core'da `TimeOnly` JSON deserialization'ı `.NET 7+`'da native destekleniyor ama format uyumu (`"HH:mm"` vs `"HH:mm:ss"`) kontrol edilmeli.

5. **`optimizationWarning` response field'ı:** Frontend bu field'ı kontrol ediyor ama backend response DTO'sunda tanımlanmamış. Backend analizinde `Result.Success()` dönüyor — warning bilgisi nasıl taşınacak? `Result<T>` ile warning message döndürülmeli veya response DTO genişletilmeli.

6. **Pre-flight summary'de "Rota otomatik optimize edilecek" bilgisi:** Kullanıcının manual DeliveryOrder düzenlemesinin üzerine yazılacağını bilmesi gerekiyor. Frontend pre-flight summary'ye bu bilgi eklenmeli.

7. **Max şoför limiti:** "Maksimum limit yok" denmiş ama 50 şoför seçilirse ne olur? Pratik bir üst limit (örn. 10) konulmalı veya en azından uyarı gösterilmeli.

## Simplifications

1. **BatchUpdateDeliveryOrderCommand'ı ayrı command olarak oluşturup sonra orchestration içinde kullanmamak** doğru yaklaşım (Architect zaten söylemiş). Ancak bu command ayrı endpoint olarak da kullanılacaksa, handler kodu DRY kalmalı — ortak bir service method extract edilebilir (ama overengineering'e kaçmamalı; ilk iterasyonda inline yeterli).

2. **Frontend `optimizationStatus` state'i:** `'idle' | 'optimizing'` yerine mevcut `isSaving` yeterli olabilir. Optimizasyon backend'de yapılıyor, frontend sadece tek bir API call yapıyor. Ayrı state gereksiz karmaşıklık ekliyor. `isSaving` + loading mesajı ("Rota optimize ediliyor ve kaydediliyor...") yeterli.

3. **DepartureTime validation:** Frontend'de `type="time"`
