# Manager Agent Analysis

## Task Summary
Şoför paneli için kapsamlı bir QR kod tabanlı sefer başlama/bitiş sistemi ve puantaj modülü geliştirilecek. Bu, mevcut sisteme birçok yeni katman ekleyen büyük bir feature. Vehicle entity'ye QR alanı eklenmesi, yeni DriverSession entity oluşturulması, cihaz güvenliği (device fingerprint), GPS tracking, atama kilitleme mekanizması, admin puantaj ekranı ve Excel export gibi cross-cutting concerns içeriyor.

## Task Type
**Major Feature** — Yeni entity'ler, schema değişiklikleri, güvenlik katmanı, cross-layer impact, external API (kamera, GPS, QR generation), admin + şoför paneli değişiklikleri.

## Affected Areas
- **Domain/Entity Layer**: Vehicle (QR alanı), yeni DriverSession entity, yeni DeviceSession entity
- **Auth/Security**: Device fingerprint, token-device binding, concurrent session prevention
- **API Layer**: Yeni endpoint'ler (QR scan, session start/end, puantaj, QR generation)
- **Business Rules**: Atama kilitleme, QR doğrulama, yan şoför akışı
- **Frontend (Şoför)**: Kamera/QR okuma, GPS capture, sefer durumu UI
- **Frontend (Admin)**: QR oluşturma/yazdırma, puantaj ekranı, Excel export
- **Database**: Migration (Vehicle alter, yeni tablolar, indexler)

## Selected Agents
- **Business Analyst** → İş akışlarının detaylı analizi, edge case'ler, kullanıcı senaryoları
- **Backend** → Entity tasarımı, API, business rules, güvenlik, kilitleme mekanizması
- **Frontend** → QR okuma (kamera), GPS, UI/UX, admin ekranları, PWA kısıtları
- **Architect** → Schema değişikliği, cross-layer impact, güvenlik mimarisi, device fingerprint stratejisi
- **QA-Reviewer** → Risk analizi, edge case validation, güvenlik açıkları

## Execution Plan
1. **Business Analyst** → İş gereksinimlerini derinlemesine analiz et, eksiklikleri tespit et
2. **Backend + Frontend** → (Paralel) BA analizini dikkate alarak plan çıkar

4. **Architect** → Çelişkileri çöz, mimari kararları finalize et

# 1. Business Analyst Agent

## İş Gereksinimi Analizi

### 1.1 Kullanıcı Personaları ve Akışları

**Persona 1: Ana Şoför (IsPrimary=true)**
- Sabah depoya gelir → Uygulamayı açar (JWT ile giriş) → Aracına gider → QR okutarak sefere başlar → Dağıtım yapar → Geri döner → QR okutarak seferi bitirir

**Persona 2: Yan Şoför (IsPrimary=false)**
- Aynı akış, ama ZonePreparationDrivers üzerinden aynı araca/zone'a atanmış durumda
- Kendi adına bağımsız DriverSession oluşturur

**Persona 3: Admin**
- Araçlara QR oluşturur/yazdırır
- Puantaj raporlarını görüntüler, filtreler, export eder
- Aktif seferleri izler (opsiyonel ama önemli)

### 1.2 Tespit Edilen Eksiklikler ve Belirsizlikler

**EKSİKLİK 1: Bir araca aynı anda birden fazla şoför QR okutabilir mi?**
- ZonePreparationDrivers tablosunda aynı araca 1 primary + N secondary şoför atanabiliyor
- İş kuralı: Aynı araca aynı zaman diliminde birden fazla aktif DriverSession olabilmeli (primary + secondary)
- **Öneri**: VehicleId + DriverId + Status=Open kombinasyonu unique olmalı, ama aynı VehicleId'de birden fazla Open session olabilmeli

**EKSİKLİK 2: Şoför QR okuttuğunda hangi ZonePreparation/Shipment ile eşleşecek?**
- QR sadece araç ID veriyor, ama şoförün o araçla hangi zone/shipment'a atandığını bilmemiz lazım
- Şoför birden fazla aktif atamaya sahip olabilir mi?
- **Öneri**: QR okutma anında şoförün o araçla aktif ataması (ZonePreparation veya Shipment) otomatik bulunmalı. Birden fazla varsa şoföre seçtirilmeli.

**EKSİKLİK 3: Sefer bitiş QR'ı illa aynı araç mı olmalı?**
- Evet olmalı — DriverSession.VehicleId ile okunan QR'daki VehicleId eşleşmeli
- Farklı araç QR'ı okutulursa hata verilmeli

**EKSİKLİK 4: GPS alınamazsa ne olacak?**
- PWA'da GPS izni reddedilebilir veya konum alınamayabilir
- **Öneri**: GPS zorunlu olmalı, alınamazsa sefer başlatılamaz. Kullanıcıya net hata mesajı gösterilmeli.

**EKSİKLİK 5: Şoför seferi bitirmeden uygulamayı kapatırsa?**
- DriverSession Open kalır, atama kilitli kalır
- **Öneri**: Admin'e "zorla kapat" yetkisi verilmeli. Opsiyonel: 24 saat sonra otomatik uyarı.

**EKSİKLİK 6: Device fingerprint değişirse (tarayıcı güncelleme, cache temizleme)?**
- Şoför kendi kendine kilitlenebilir
- **Öneri**: Admin "cihaz sıfırla" butonu olmalı. Fingerprint değişimi durumunda re-login yeterli olmalı (yeni fingerprint'e bind).

**EKSİKLİK 7: Puantaj "toplam süre" nasıl hesaplanacak?**
- EndTime - StartTime = TotalDurationMinutes
- Ama gece yarısını geçen seferler olabilir mi? (Olası değil ama handle edilmeli)
- Mola süresi düşülecek mi? (İş kuralı: şimdilik hayır, brüt süre)

**EKSİKLİK 8: QR okutma sırası**
- Ana şoför mü önce okutmalı yoksa sıra önemli değil mi?
- **Öneri**: Sıra önemli değil. Her şoför bağımsız session açar.

**EKSİKLİK 9: Offline durum**
- PWA olarak internet kesilirse QR okutma ne olacak?
- **Öneri**: v1'de offline desteksiz. İnternet zorunlu. İleride offline-first eklenebilir.

### 1.3 İş Kuralları Matrisi

| Durum | Kural |
|-------|-------|

**Yeni: DriverSession Entity:**
```csharp
public class DriverSession : BaseAuditableEntity
{
    public Guid Id { get; private set; }
    public Guid DriverId { get; private set; }
    public Guid VehicleId { get; private set; }
    public Guid? ZonePreparationId { get; private set; }  // nullable - birden fazla atama senaryosu
    public Guid? ShipmentId { get; private set; }          // hangi shipment ile ilişkili
    
    public DateTime StartTime { get; private set; }
    public double StartLatitude { get; private set; }
    public double StartLongitude { get; private set; }
    
    public DateTime? EndTime { get; private set; }
    public double? EndLatitude { get; private set; }
    public double? EndLongitude { get; private set; }
    
    public int? TotalDurationMinutes { get; private set; }
    public DriverSessionStatus Status { get; private set; }
    
    public string? ClosedByAdminId { get; private set; }   // admin zorla kapattıysa
    public string? Notes { get; private set; }
    
    // Navigation
    public Driver Driver { get; private set; }
    public Vehicle Vehicle { get; private set; }
    public ZonePreparation? ZonePreparation { get; private set; }
    
    // Factory method
    public static DriverSession StartNew(
        Guid driverId, Guid vehicleId, Guid? zonePreparationId, Guid? shipmentId,
        double latitude, double longitude)
    {
        return new DriverSession
        {
            Id = Guid.NewGuid(),
            DriverId = driverId,
            VehicleId = vehicleId,
            ZonePreparationId = zonePreparationId,
            ShipmentId = shipmentId,
            StartTime = DateTime.UtcNow,
            StartLatitude = latitude,
            StartLongitude = longitude,
            Status = DriverSessionStatus.Open
        };
    }
    
    public void Close(double latitude, double longitude)
    {
        EndTime = DateTime.UtcNow;
        EndLatitude = latitude;
        EndLongitude = longitude;
        TotalDurationMinutes = (int)(EndTime.Value - StartTime).TotalMinutes;
        Status = DriverSessionStatus.Closed;
    }
    
    public void ForceClose(string adminId, string? notes = null)
    {
        EndTime = DateTime.UtcNow;
        TotalDurationMinutes = (int)(EndTime.Value - StartTime).TotalMinutes;
        Status = DriverSessionStatus.ForceClosed;
        ClosedByAdminId = adminId;
        Notes = notes;
    }
}

public enum DriverSessionStatus
{
    Open = 0,
    Closed = 1,
    ForceClosed = 2
}
```

## 1. GEREKSİNİM ANALİZİ

### 1.1 İş Gereksinimi Yeterliliği

**Tespit:** Gereksinimler operasyonel akışı genel hatlarıyla iyi tanımlıyor, ancak **sahada yaşanacak gerçek hayat senaryoları** büyük ölçüde eksik. Bu bir depo/lojistik operasyonu — teknoloji değil, saha gerçekliği belirleyici.

**Neden önemli:** Yazılım ekibi "happy path"i geliştirir ama şoförler sabah 05:00'te, soğukta, aceleyle, bazen düşük internet hızıyla bu sistemi kullanacak. Kabul oranını düşürecek her sürtünme noktası projeyi başarısız kılar.

**Öneri:** Aşağıdaki tüm senaryolar tanımlanıp iş kuralları netleştirilmeden geliştirmeye başlanmamalı.

---

### 1.2 Kullanıcı Grupları ve Etki Analizi

| Kullanıcı | Doğrudan Etkilenen | Dolaylı Etkilenen |
|---|---|---|
| **Ana Şoför** | Günlük iş akışı tamamen değişiyor | — |
| **Yan Şoför** | Bağımsız session açma yeni bir kavram | — |
| **Depo Operasyon Amiri** | — | Sabah atamaları artık kilitlenebilir, esneklik azalır |

**Neden önemli:** Kilitleme mekanizması bu kişinin iş akışını doğrudan etkiler. "QR okutuldu, değiştiremiyorum" durumu sabah operasyonu tıkayabilir.

**Öneri:** Operasyon amirine "kilitli atamayı override etme" yetkisi tanımlanmalı. Admin'den ayrı, daha dar kapsamlı bir yetki.

## 2. EKSİK GEREKSİNİMLER VE KRİTİK SENARYOLAR

### 2.1 🔴 Kritik: "Ya Şoför Yanlış Araca QR Okuttursa?"

**Tespit:** Şoför A, Araç-1'e atanmış ama yanlışlıkla yanındaki Araç-2'nin QR'ını okutuyor. Sistem ne yapacak?

**Neden önemli:** Depoda araçlar yan yana park ediyor. QR sticker'lar birbirine yakın. Bu hata **kesinlikle olacak**.

**Öneri:**
- QR okutma sonrası araç plakasını büyük fontla göster: **"34 ABC 123 aracıyla sefer başlatılacak. Onaylıyor musunuz?"**
- Şoför ile QR'daki araç arasında aktif atama yoksa: **"Bu araca atamanız bulunmuyor"** hatası
- Onay adımı olmadan session açılmamalı

---

### 2.2 🔴 Kritik: "Şoför QR Okuttu Ama Aslında Depoya Dönmeyecek — Direkt Eve Gidecek"

**Tespit:** Sefer bitiş QR'ı "aynı aracın QR'ını tekrar okutur" diyor. Ama şoför dağıtımı bitirdikten sonra aracı depoya mı bırakıyor, yoksa araçla mı eve gidiyor?

**Neden önemli:** Eğer araç depoya dönüyorsa, bitiş QR'ı depoda okutulur — mantıklı. Ama şoför araçla eve gidiyorsa, bitiş QR'ını ne zaman, nerede okutacak?

**Öneri:**
- İki senaryo da desteklenmeli
- Bitiş QR'ı aracın yanında olmayı gerektiriyorsa → bu bir kısıt, şoföre bildirilmeli
- Alternatif: **QR'sız manuel bitiş butonu** (GPS ile) ama bu güvenliği düşürür
- **İş kararı gerekli:** Bitiş QR'ı zorunlu mu, yoksa "QR veya manuel buton" mu?

---

### 2.3 🔴 Kritik: "Aynı Şoför Gün İçinde Birden Fazla Sefer Yapabilir mi?"

**Tespit:** Gereksinimde "bir şoför aynı anda bir Open session olabilir" var ama **gün içinde ardışık seferler** senaryosu tanımlanmamış.

**Neden önemli:** Bazı operasyonlarda şoför sabah bir sefer, öğleden sonra başka bir sefer yapabilir. Hatta farklı araçla bile olabilir.

**Öneri:**
- Bir şoförün aynı gün içinde birden fazla kapalı (Closed) session'ı olabilmeli
- Puantajda günlük toplam süre = tüm session'ların toplamı olarak hesaplanmalı
- Bu puantaj ekranında net gösterilmeli: "Sefer 1: 06:00-10:30 | Sefer 2: 13:00-16:45 | Toplam: 8 saat 15 dk"

---

### 2.4 🟡 Önemli: "QR Sticker Yırtılırsa / Okunamaz Hale Gelirse?"

**Tespit:** Fiziksel QR sticker zamanla yıpranır, kirlenir, yırtılır. Araçlar yıkanır, güneşe maruz kalır.

**Neden önemli:** Bir sabah şoför QR okutamayınca tüm operasyon durur.

**Öneri:**
- Admin panelinde **"QR Yenile"** butonu (yeni QR = aynı içerik, sadece QrGeneratedAt güncellenir)
- Her aracın QR'ı her zaman tekrar yazdırılabilir olmalı
- Opsiyonel: Şoför uygulamasında **araç plakasını manuel girerek** sefer başlatma (acil durum modu, admin onayı gerektiren)
- QR sticker malzemesi önerisi: **laminasyonlu, suya dayanıklı** olmalı — bu teknik değil operasyonel karar

---

### 2.5 🟡 Önemli: "İnternet Kesilirse Ne Olacak?"

**Tespit:** v1'de offline desteksiz denmiş. Ama lojistikte internet her zaman güvenilir değil — bodrum kat depo, dağ köyü teslimat, tünel geçişi.

**Neden önemli:** Şoför QR'ı okutuyor, internet yok, sefer başlamıyor. Bekliyor. Operasyon gecikiyor.

**Öneri:**
- v1'de gerçekten offline çalışmak gerekmez, **ama şu olmalı:**
  - QR okutma ve GPS alımı offline yapılabilmeli
  - Veri **local'de tutulmalı**, internet gelince **otomatik senkronize edilmeli**
  - Bu "offline-first" değil, "connectivity-resilient" yaklaşım — çok daha az eforlu
- Eğer bu da v1'de yoksa, en azından **net hata mesajı + retry butonu** olmalı
- "İnternet bağlantınız yok, bağlantı sağlandığında tekrar deneyin" — bu mesaj yetmez, şoför 5 dakika bekleyemez

---

### 2.6 🟡 Önemli: "Device Fingerprint Değişince Ne Olacak?"

**Tespit:** PWA'da device fingerprint güvenilir değil. Tarayıcı güncellemesi, cache temizleme, private mode, farklı tarayıcı kullanma — hepsi fingerprint'i değiştirir.

**Neden önemli:** Şoför "ben hiçbir şey yapmadım" diyecek ama sistem onu tanımayacak. Sabah 05:00'te admin'i arayacak. Admin telefonu açmayacak.

**Öneri:**
- Device fingerprint'i **hard block** olarak değil, **soft verification** olarak kullan
- Fingerprint değişirse → şoföre SMS/OTP ile doğrulama gönder → yeni cihazı kaydet
- Ya da → şoför re-login yaptığında yeni fingerprint otomatik kabul edilsin
- **Kesinlikle yapılmaması gereken:** Fingerprint değişiminde hesabı kilitleme ve sadece admin müdahalesiyle açma — bu operasyonu öldürür

---

### 2.7 🟡 Önemli: "Puantaj Verisi Kimin İçin?"

**Tespit:** Puantaj ekranı tanımlanmış ama **bu veriyi kim, ne için kullanacak** belirsiz.

**Neden önemli:** Puantajın amacı belirler: maaş hesaplama mı, performans takibi mi, yasal zorunluluk mu?

**Öneri:**
- **Maaş hesaplama ise:** Dakika düzeyinde doğruluk kritik, session'lar düzenlenemez olmalı (audit trail), admin düzeltme yapabilmeli ama log tutulmalı
- **Performans takibi ise:** Karşılaştırma metrikleri lazım (ortalama süre, en hızlı, en yavaş)
- **Yasal ise:** Veri saklama süresi, imzalatma/onay mekanizması gerekebilir
- **İş kararı gerekli:** Puantajın birincil amacı nedir?

---

### 2.8 🟢 Düşük Öncelik Ama Sorulmalı

**"Birden fazla araçla çalışan şoför var mı?"**
- Sabah Araç-1 ile sefer, öğlen Araç-2 ile sefer?
- Varsa sistem buna izin vermeli (farklı session'lar, farklı araçlar)

**"Şoför araç değiştirmek zorunda kalırsa?"**
- Araç bozuldu, yarı yolda başka araçla devam edecek
- Mevcut session kapatılıp yeni session açılmalı mı?
- Yoksa admin müdahalesi mi gerekli?

**"QR oluşturulmuş ama sticker yapıştırılmamış araçlar?"**
- QR'ı olan ama fiziksel olarak sticker'ı olmayan araç problemi
- Envanter/checklist önerilir

## 3. ALTERNATİF YAKLAŞIMLAR

### 3.1 QR Yerine NFC

**Tespit:** NFC tag'ler QR sticker'dan daha dayanıklı ve hızlı.

**Neden düşünülmeli:** Yıpranma yok, kamera açmaya gerek yok, dokunmak yeterli.

**Neden reddedilmeli:** PWA'da NFC desteği kısıtlı (sadece Chrome Android). iOS desteklemez. QR daha evrensel. **QR doğru seçim.**

### 3.2 QR Yerine Manuel Araç Seçimi

**Tespit:** Şoföre atanmış araçları dropdown'da göster, seçsin.

**Neden reddedilmeli:** QR'ın amacı **fiziksel doğrulama** — şoför gerçekten o aracın yanında olduğunu kanıtlıyor. Dropdown bunu sağlamaz. **QR doğru seçim.**

### 3.3 Device Fingerprint Yerine SMS OTP

**Tespit:** Her login'de SMS doğrulama, cihaz bağlama yok.

**Değerlendirme:** Daha basit, daha güvenilir, ama her giriş için SMS maliyeti. Günde 1 login ise makul. **Hibrit öneri:** İlk giriş SMS OTP + cihaz kayıt, sonraki girişlerde fingerprint check. Fingerprint değişirse tekrar OTP.

### 3.4 Puantaj İçin Basitleştirilmiş v1

**Tespit:** Tam puantaj ekranı yerine, sadece **session listesi + tarih filtresi** yeterli olabilir v1 için.

**Öneri:** Excel export'u v1.1'e bırak. v1'de sadece listeleme + filtreleme. Muhasebe ihtiyacı acilse, admin doğrudan DB'den veya basit bir rapor çekebilir.

    public Driver Driver { get; private set; } = null!;
    public Vehicle Vehicle { get; private set; } = null!;

    public static DriverSession Create(
        Guid driverId,
        Guid vehicleId,
        Guid? zonePreparationId,
        Guid? shipmentId,
        double latitude,
        double longitude)
    {
        return new DriverSession
        {
            Id = Guid.NewGuid(),
            DriverId = driverId,
            VehicleId = vehicleId,
            ZonePreparationId = zonePreparationId,
            ShipmentId = shipmentId,
            StartTime = DateTime.UtcNow,
            StartLatitude = latitude,
            StartLongitude = longitude,
            Status = DriverSessionStatus.Open
        };

    public void Close(double latitude, double longitude)
    {
        Guard.Against.InvalidOperationIf(
            Status != DriverSessionStatus.Open,
            "Sadece açık session kapatılabilir.");

        EndTime = DateTime.UtcNow;
        EndLatitude = latitude;
        EndLongitude = longitude;
        TotalDurationMinutes = (int)(EndTime.Value - StartTime).TotalMinutes;
        Status = DriverSessionStatus.Closed;
    }

    public void ForceClose(string adminId, string? notes = null)
    {
        Guard.Against.InvalidOperationIf(
            Status != DriverSessionStatus.Open,
            "Sadece açık session zorla kapatılabilir.");

        EndTime = DateTime.UtcNow;
        TotalDurationMinutes = (int)(EndTime.Value - StartTime).TotalMinutes;
        Status = DriverSessionStatus.ForceClosed;
        ClosedByAdminId = adminId;
        Notes = notes;
    }

# İş Analisti Değerlendirmesi — QR Sefer & Puantaj Modülü

---

## 1. Gereksinim Analizi

### 1.1. İş Gereksinimi Yeterli mi?

**Tespit:** İş gereksinimi genel hatlarıyla tutarlı ancak birkaç kritik noktada "sahada ne olacak?" sorusu yanıtsız bırakılmış. Teknik plan oldukça sağlam bir mimari sunuyor ama bazı iş senaryolarını varsayımla geçiyor.

**Neden önemli:** Teknik ekip varsayımla kodlamaya başlarsa, sahadan gelen ilk geri bildirimde revizyon kaçınılmaz olur. Şimdi 2 saat düşünmek, sonra 2 hafta refactor'ı önler.

---

## 2. Kritik Eksik İş Kuralları

### 2.1. "Şoför Yanlış Araca QR Okuttu" Senaryosu

**Tespit:** Backend planında atama kontrolü var: "Şoför bu araca atanmış mı?" Ama sahada çok yaygın bir durum atlanmış — **araç değişikliği**. Sabah planlama yapılır, araç A atanır. Araç A bozulur, şoför araç B'ye bindirilir. Ama sistemde hâlâ araç A ataması var.

**Neden önemli:** Bu senaryo lojistikte "her gün" yaşanır. Araç arızası, lastik patlaması, kaza, bakım — günlük olay. Şoför sahada araç B'nin QR'ını okuttuğunda sistem "Bu araca atamanız yok" diyecek. Şoför sefere çıkamayacak. Dispatch çağrılacak, atama güncellenecek, zaman kaybedilecek.

**Öneri:**
- **Acil araç değişikliği akışı** tanımlanmalı: Admin/dispatcher sahadan gelen taleple hızlıca atamayı güncelleyebilmeli
- Veya: Şoföre "Atamanız farklı araçtayken devam etmek istiyor musunuz?" uyarısı gösterilip, override imkanı sunulmalı (log'lanarak)
- Minimum viable: En azından hata mesajı "Lütfen yöneticinizi arayarak atama güncellenmesini isteyin" şeklinde yönlendirici olmalı

---

### 2.2. "Şoför QR Okutmayı Unuttu" Senaryosu

**Tespit:** Backend planında ForceClose var ama **"sefer başlatılmadan dağıtıma çıkıldı"** durumu hiç ele alınmamış. Şoför acele eder, QR okutmadan yola çıkar. Dağıtımı yapar, döner, bitiş QR'ı okutmak ister ama açık session yoktur.

**Neden önemli:** Puantaj sisteminin güvenilirliği doğrudan buna bağlı. Eğer şoförlerin %20'si QR okutmayı unutursa, puantaj verisi güvenilmez olur ve sistem amacına ulaşamaz.

**Öneri:**
- **Admin retroaktif session oluşturma:** "Şoför X, araç Y, saat 08:00-17:00 arası sefer yaptı" şeklinde elle kayıt girebilmeli
- **Hatırlatma mekanizması:** Şoför paneline giriş yaptığında, atanmış ama QR okutmamışsa belirgin uyarı banner'ı gösterilmeli: "Seferinizi başlatmayı unutmayın!"
- **Sefer başlatmadan rota görüntüleme:** Mevcut `/driver/route` endpoint'i açık session olmadan da çalışmaya devam etmeli. QR okutma zorunluluk değil, kayıt mekanizması olmalı. Yoksa şoför işini yapamaz

---

### 2.3. "Sefer Bitmeden Uygulama Kapandı" Senaryosu

**Tespit:** Şoför QR okutur, sefer başlar. Sonra telefon kapanır (şarj biter), uygulama crash olur, veya şoför uygulamayı yanlışlıkla siler. Geri geldiğinde ne olacak?

**Neden önemli:** PWA uygulamada bu çok olası. Native app gibi background service çalıştıramıyorsunuz.

**Öneri:**
- Backend planındaki `GetActiveDriverSessionQuery` bu durumu kısmen çözüyor — şoför tekrar giriş yapınca aktif session'ı görecek. Bu iyi.
- Ama şarj biten/telefonu kaybolan şoför o gün geri gelemezse ne olacak? **Otomatik session temizleme** kuralı lazım: Gece 03:00'te hâlâ açık olan session'lar otomatik olarak `ForceClosed` yapılmalı, notes alanına "Otomatik kapatma — 24 saat aşımı" yazılmalı
- Puantaj raporunda bu session'lar farklı renk/ikon ile işaretlenmeli ki admin "bu veri güvenilir mi?" sorusunu cevaplayabilsin

---

### 2.4. Yan Şoför (IsPrimary=false) Akışı Detaylandırılmamış

**Tespit:** İş gereksiniminde "Yan şoför de aynı işlemi yapar" yazıyor ama **yan şoförün seferi ana şoförden bağımsız mı?** Bu çok kritik bir soru.

**Neden önemli:** Aynı araçta 2 şoför var. İkisi de QR okutursa 2 ayrı DriverSession açılır. Puantajda bu araç için 2 şoför eş zamanlı çalışmış görünür. Doğru. Ama:

- Birisi bitiş QR'ı okutmazsa ne olur? 
- Ana şoför bitirir ama yan şoför unutursa?
- Birisi seferi başlatır diğeri başlatmazsa?

**Öneri:**
- Kurumsal karar gerekiyor: **Yan şoförün QR okutması zorunlu mu?** 
- Eğer zorunluysa: Ana şoför seferi bitirdiğinde yan şoföre de hatırlatma gitmeli
- Eğer zorunlu değilse: Sadece ana şoförün session'ı yeterli, yan şoför puantaja ana şoförün session'ı üzerinden dahil olmalı
- **Önerim:** İlk versiyonda sadece ana şoför (IsPrimary=true) QR okutsun. Yan şoför aynı araca atanmış olduğu için puantajda "aynı session'ın yan şoförü" olarak otomatik işaretlensin. Böylece iki kişi için iki ayrı QR okutma beklentisi olmaz — sahada karmaşıklık azalır.

---

### 2.5. Aynı Gün Birden Fazla Sefer

**Tespit:** Bir şoför sabah bir dağıtım yapar, döner, öğleden sonra ikinci bir dağıtıma çıkar. Yani aynı gün aynı şoför 2-3 sefer yapabilir.

**Neden önemli:** Backend planında "şoförün aktif Open session'ı var mı" kontrolü var, bu iyi — eş zamanlı session engellenmiş. Ama puantaj raporlamasında bu nasıl görünecek? Günlük toplam mı yoksa sefer bazlı mı?

**Öneri:**
- Puantaj ekranında **hem sefer bazlı detay hem günlük toplam** gösterilmeli
- Raporlama DTO'suna `DailyTotalMinutes` gibi bir aggregate alan eklenmeli
- Excel export'ta her satır bir sefer olmalı, en altta günlük/aylık toplam olmalı

## 3. Cihaz Güvenliği — Gerçekçilik Değerlendirmesi

### 3.1. Device Fingerprint Güvenilirliği

**Tespit:** IMEI kullanılamıyor (tarayıcı kısıtı, doğru karar). Device fingerprint önerilmiş. Ama tarayıcı fingerprint'i aslında **hiç güvenilir değildir.** Kullanıcı tarayıcı önbelleğini temizlerse, farklı tarayıcıyla girerse, tarayıcı güncellenirse fingerprint değişir.

**Neden önemli:** Şoför Chrome'dan giriş yapar, cache temizler, tekrar giriş yapar — sistem "yeni cihaz" sanıp önceki oturumu sonlandırır. Aslında aynı telefon, aynı kişi.

**Öneri:**
- **Beklentiyi doğru kalibre edin:** Device fingerprint, "kesin güvenlik" değil, "makul düzeyde caydırıcılık" sağlar. Bunu kabul ederek yola çıkın
- **Kullanıcı deneyimi koruması:** Fingerprint değiştiğinde oturumu hemen sonlandırmak yerine, admin'e bildirim gönderin. Şoför çalışmasına devam etsin, admin anomaliyi incellesin
- **İlk MVP'de device session'ı sadece logging amaçlı tutun**, enforce etmeyin. Veriye bakarak gerçekten çoklu cihaz kullanım problemi var mı görün. Yoksa karmaşıklık katmayın

## 4. Atama Kilitleme — Edge Case'ler

### 4.1. Kilitli Durumda Acil Müdahale

**Tespit:** "QR okutulduysa atama değişimi engellenir" kuralı teknik olarak doğru ama operasyonel olarak riskli. 

**Senaryo:** Şoför sefere çıktı (session açık), araç ortada bozuldu. Dispatcher bu aracı başka bir şoföre atamak istiyor — sistem engelliyor. Veya şoför kaza yaptı, hastaneye kaldırıldı — kimse atamayı değiştiremiyor.

**Neden önemli:** Gerçek hayatta "kural dışı" durumlar kaçınılmazdır. Sistem %100 katı olursa operasyon durur.

**Öneri:**
- **Admin override yetkisi:** Kilidi admin'in kaldırabilmesi lazım (ForceClose zaten var, iyi)
- Ama ForceClose + yeni atama + yeni QR başlatma akışı **tek bir admin ekranından** yapılabilmeli. 3 ayrı ekranda 5 ayrı işlem yapmak dispatch'i sinir eder
- **Hızlı akış:** "Bu seferi zorla kapat ve aracı serbest bırak" tek buton olmalı

---

### 4.2. QR Başlatma ile Atama Arasındaki Sıralama

**Tespit:** Backend planında atama kontrolü var: "Şoför bu araca atanmış mı?" Ama soru şu — **atama her zaman QR okutmadan önce mi yapılır?**

**Senaryo:** Bazen sahada spontane kararlar alınır. Dispatch "sen şu aracı al" der, şoför QR okutur, atama sonradan sisteme girilir.

**Öneri:**
- Eğer operasyonel süreç her zaman "önce atama, sonra QR" ise → mevcut kontrol doğru
- Eğer bazen tersi oluyorsa → QR okutma sırasında "atama bulunamadı ama session oluşturulsun mu?" gibi bir fallback gerekir
- **Kurumsal karar gerektiriyor:** Operasyon müdürüne sorun. Benim önerim: İlk versiyonda katı tutuluğa bir "admin atama ile QR'ı eşleştir" özelliği eklensin

---

### 5.1. Puantaj Neyin Ölçüsü?

### Adım 2.1 — Device Fingerprint Generation (Frontend Utility)

```
Dosya: src/utils/deviceFingerprint.js (Şoför paneli frontend)
```

```javascript
// Basit ama yeterli fingerprint — IMEI olmadığı için:
export async function generateDeviceFingerprint() {
  const components = [
    navigator.userAgent,
    navigator.language,
    screen.width + 'x' + screen.height,
    screen.colorDepth,
    new Date().getTimezoneOffset(),
    navigator.hardwareConcurrency || 'unknown',
    navigator.platform
  ];
  
  const raw = components.join('|');
  const encoder = new TextEncoder();
  const data = encoder.encode(raw);
  const hashBuffer = await crypto.subtle.digest('SHA-256', data);
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  return hashArray.map(b => b.toString(16).padStart(2, '0')).join('');
}
```

**Not**: Bu fingerprint %100 unique değildir. Aynı model cihaz + aynı tarayıcıda çakışma olabilir. Ama IMEI olmadan pragmatik çözüm budur. Login'de `localStorage`'a ek bir random UUID (`deviceInstanceId`) yazılarak güçlendirilmeli:

```javascript
export function getOrCreateDeviceInstanceId() {
  let id = localStorage.getItem('device_instance_id');
  if (!id) {
    id = crypto.randomUUID();
    localStorage.setItem('device_instance_id', id);
  }
  return id;
}

export async function getFullDeviceId() {
  const fingerprint = await generateDeviceFingerprint();
  const instanceId = getOrCreateDeviceInstanceId();
  return `${fingerprint}_${instanceId}`;
}
```

### Adım 2.2 — Login Flow'a Device Binding

### Adım 2.3 — Device Validation Middleware

```
Dosya: Infrastructure/Middleware/DeviceValidationMiddleware.cs
```

- Sadece `/driver/*` endpoint'lerine uygula (admin endpoint'leri hariç)
- Her request'te:
  1. JWT'den UserId al
  2. Request header'dan `X-Device-Fingerprint` al
  3. DB'den aktif DeviceSession bul (UserId + IsActive=true)
  4. DeviceFingerprint eşleşmiyorsa → **401 Unauthorized** + `{ code: "DEVICE_MISMATCH", message: "Oturumunuz başka bir cihazdan açıldı" }`
  5. DeviceSession yoksa → **401** + `{ code: "NO_ACTIVE_SESSION" }`

**Performans**: Her request'te DB sorgusu maliyetli. Çözüm:
- Redis/Memory cache ile aktif device session'ları cache'le (userId → fingerprint)
- Cache TTL: 5 dakika
- Login/logout'ta cache invalidate et
- Eğer Redis yoksa → In-memory `IMemoryCache` yeterli (tek sunucu varsayımı)

```
Dosya: Frontend — API interceptor (axios/fetch wrapper)
```

- Her request'e `X-Device-Fingerprint` header'ı ekle
- 401 + `DEVICE_MISMATCH` gelirse → kullanıcıyı login sayfasına yönlendir + "Oturumunuz başka cihazdan açıldı" mesajı göster
